# CLAUDE.md

## プロジェクト概要

このリポジトリは AI ネイティブ開発方法論に基づいて運用されています。
`.ai-native/` ディレクトリにフェーズ定義・レビュー基準・ロール定義があります。

### Source of Truth（SoT）宣言

| 領域 | SoT | 説明 |
|------|-----|------|
| AI ネイティブ開発方法論全体 | `.ai-native/methodology/` | ロール定義・フェーズ定義・レビュー基準・運用テンプレートの正規定義 |
| Claude Code 固有の実装ルール | この `CLAUDE.md` | Push 前チェック、バグ修正手順、技術スタック別確認ポイント等の Claude Code 環境固有ルール |

- 方法論全体の原則（最上位原則、構造原則、レビュー基準等）は `.ai-native/methodology/` 配下のドキュメントが正
- 本ファイルの「開発原則」は Claude Code 環境でのコーディング作業に特化した実装ルールであり、方法論の原則を補完する
- 両者に矛盾がある場合は `.ai-native/methodology/` 側の定義を優先する

## 開発原則

以下の原則は全ての開発作業に適用する（個別の適用条件がある場合はその条件に従う）。Push 前に必ず自問し、違反がないことを確認すること。

### 1. 手動ステップを残さない

ユーザーに手順書を読ませて手動操作させる設計は負債。自動化できる初期化・セットアップ処理は、コード側で完結させる。手動手順はフォールバック・更新用としてのみ残す。

### 2. 冪等性と状態保護

再実行可能な設計にする。ただし「何度実行しても壊れない」だけでは不十分。**再実行で既存の進捗・ログ・ステータスが巻き戻る副作用を許容しない**。記録系データは保護し、設定系データのみ更新する。

### 3. 既存パターンの再利用

新しいユーティリティや抽象を作る前に、既存コードに同じパターンがないか探す。既にある関数・ヘルパーで解決できるなら、それを使う。不必要な新規コードを書かない。

### 4. 非ブロッキングなエラーハンドリング

補助的な処理（ラベル作成、Webhook登録など）の失敗が、主要なフロー全体を止めてはならない。「できたところまで進めて結果を報告する」グレースフルデグラデーションを原則とする。致命的な失敗のみ例外を投げる。

### 5. コードとドキュメントの一貫性

コードの振る舞いを変更したら、関連するドキュメントを**全件チェック**して更新する。コードだけ直してドキュメントが旧情報のまま残る状態は「未完了」。テンプレート版とルート版など、同一であるべきファイルは差分検証する。

### 6. データフロー整合性（I/Fファースト）

複数のデータストアを持つシステムでは、Source of Truth (SoT) を明確にし、データフローの一貫性を保証する。

#### SoT の原則
- 各データの SoT がどのシステムかを明示的に宣言する。SoT が不明確なデータは同期バグの温床になる。
- SoT 側への書込を先に行い、キャッシュ側はその結果を反映する形で更新する。逆順は不整合の原因。
- SoT から復元できないデータが存在する場合、それを設計上の判断として文書化する。

#### 変更時の必須確認
1. **新しいデータストア書込を追加する場合:** そのデータの SoT はどちらか？ SoT への書込が先、キャッシュ更新が後の順序になっているか？
2. **外部システム連携を追加する場合:** イベント受信ハンドラ（Webhook 等）と手動回復パス（再同期機能）の両方が存在するか？
3. **イベントトリガー処理を追加する場合:** 冪等性が保証されているか？ 失敗が主要フローをブロックしないか？
4. **新しいエンティティを追加する場合:** 型定義・アクセス制御ルール・SoT 宣言を全て同時に更新しているか？

### 7. 下位互換性とデータ保護

既存のインターフェースやデータ構造を変更する場合は、**下位互換性への影響を必ず評価する**。既存データに影響を与える変更がやむを得ない場合は、データ更新パッチを生成し、変更内容と操作手順をオペレーターに説明する。「壊れない」だけでなく「既存のデータ・利用者が困らない」ことを確認する。

> **原則2（冪等性と状態保護）との関係:** 原則2は「再実行で既存データが巻き戻らないこと」に焦点を当てる。本原則はより広く「スキーマ変更・I/F変更が既存データやクライアントを破壊しないこと」をカバーする。原則6（データフロー整合性）の変更時確認と合わせて適用する。

### 8. レスポンシブ対応

> **適用条件:** Web/モバイル UI を持つプロジェクトの場合に適用する。CLI・バッチ・API のみのプロジェクトでは本原則はスキップする。

UI を実装・変更する際は、**モバイル表示を前提としたレスポンシブデザインを常に組み込む**。PC 向けにリスト・テーブルで表示するデータは、モバイルではカード型レイアウト等の可読性の高い形式を検討する。「PC で動く」だけでは完了としない。

> **方法論との関係:** 方法論の USABILITY_STANDARDS（U-2: 出力の直感性、U-5: アクセシビリティ配慮）を補完する Claude Code 固有の実装ルールである。

### 9. 改修後の反復レビュー

コードの修正・追加を行った後は、**独立したロール（コードレビュアー・システム監査官）によるレビューと監査を繰り返し実施し、指摘事項がゼロになるまでイテレーションを回す**。セルフチェック（Push 前チェック）は自己レビューであり、独立ロールによるレビューの代替にはならない。1回のレビューで見つけた問題を修正した結果、新たな問題が生じていないかを必ず再確認する。「直した」で終わらず「直した結果も問題ない」まで確認して完了とする。

> **方法論との関係:** 本原則は `.ai-native/methodology/common/core-principles.md` の SP-8（インクリメンタルレビューパイプライン）を Claude Code 環境で実践するための実装ルールである。SP-8 の詳細定義が SoT。

### Push 前の最終セルフチェック

コミット・Push する前に、以下を自問する。1つでも「いいえ」があれば修正してから Push する。

1. **手動ステップ:** この変更で、ユーザーに新たな手動操作を要求していないか？
2. **冪等性:** この処理を2回実行したら、何が起きるか？ 既存データが破壊されないか？
3. **既存コード:** 同じことをする関数やパターンが既にないか？ 重複を作っていないか？
4. **エラーハンドリング:** 補助処理の失敗で全体が止まる箇所はないか？ 想定エラーにエラーコードが付与されているか？
5. **ドキュメント:** この変更で記述が古くなるドキュメントはないか？ 全件確認したか？
6. **波及範囲:** 変更の影響を推測ではなく、Grep/検索で網羅的に確認したか？
7. **データフロー:** SoT → キャッシュの方向が守られているか？ 同期パスに欠落がないか？
8. **下位互換性:** 既存データ・既存 I/F に影響があるか？ ある場合、データ更新パッチを用意し、オペレーターへの説明を行ったか？
9. **レスポンシブ:** UI の変更がある場合、モバイル表示を確認したか？（UI を持つプロジェクトの場合）
10. **反復レビュー:** レビュー・監査のイテレーションを回し、未解決の指摘がゼロになったか？

## バグ修正の手順

1. エラーの全フローをエンドツーエンドでトレースし、関連ファイルを **全て** 読む
2. 技術スタック固有の確認ポイント（後述）を照合する
3. 確認ポイントにない問題、または確信が持てない場合は **WebFetch で公式ドキュメントを確認してから** 修正案を出す
4. 修正案を出した後「この修正で残る問題はないか」を自問する
5. 複数の問題がある場合は **全てまとめて1回で修正する**（1つ見つけた時点で飛びつかない）

## 技術スタック別の確認ポイント

### Firestore
- Security Rules: list（クエリ）と get（単一取得）でルール評価が異なる。list では `exists()` / `get()` がクエリ制約から検証不可
- バッチ書込: `get()` は書込前の状態を参照。`getAfter()` で書込後を参照
- クエリ: ルールの条件がクエリ制約から検証可能か確認

### Nuxt
- `.client.ts` プラグインは SSR 時にサーバーで実行されない
- ミドルウェアは SSR 時にサーバーでも実行される
- `ssr: false` にしない限り、サーバー/クライアント両方の動作を考慮

### Vue
- `useState` / `ref` に格納するオブジェクトは Proxy traverse 安全か確認
- Firebase User 等の外部オブジェクトは直接格納しない（plain object に変換）

### AWS
- IAM ポリシー: Action / Resource / Condition の組み合わせ。最小権限の原則
- Lambda: コールドスタート、タイムアウト、メモリ設定の影響を考慮
- API Gateway: ステージ変数、CORS 設定、認可方式（Cognito / IAM / Lambda Authorizer）
- CloudFormation / CDK: デプロイ順序の依存関係。循環参照に注意
- S3: バケットポリシーと IAM ポリシーの両方が許可する必要がある
- VPC: セキュリティグループ / NACL / ルートテーブルの3層を確認

### .NET / C#
- async/await: `ConfigureAwait(false)` の要否。デッドロック（SynchronizationContext）に注意
- DI コンテナ: Scoped / Transient / Singleton のライフタイム不一致によるキャプチャードスコープ問題
- Entity Framework: N+1 クエリ、遅延読み込み vs 明示的読み込み。マイグレーションの適用順序
- Middleware パイプライン: 実行順序が登録順に依存。認証→認可→エンドポイントの順序を確認
- appsettings.json: 環境別設定（Development / Production）のオーバーライド順序

## 未知の問題に当たった場合

訓練データの知識だけで推測せず、**必ず WebFetch で公式ドキュメントの該当ページを確認する**。
特にセキュリティルール・認証フロー・SSR 挙動・リアクティビティの境界条件は、
バージョンによって挙動が変わるため、ドキュメントで裏取りしてから修正する。

## TOOL_CAPABILITY_MAPPING（ツール機能マッピング）

> **SoT:** Claude Code 固有のツール機能活用ルールは本セクションが SoT。方法論全体のツール機能マッピングは `.ai-native/methodology/common/tool-feature-mapping.md` が SoT。

Claude Code の機能を方法論の各タッチポイントで最大限に活用する。新機能のリリース時に本マッピングを更新すること。方法論全体（ツール非依存）の機能マッピングは `.ai-native/methodology/common/tool-feature-mapping.md` を参照。本セクションは Claude Code 固有の実装詳細に特化する。

| 機能 | 方法論上の活用ポイント | 使い方 |
|------|---------------------|-------|
| 可視化（Mermaid レンダリング） | DOCUMENTATION_VISUALIZATION_RULES 準拠の全ロール出力 | アーキテクチャ図、データフロー、フェーズ遷移図の生成 |
| サブエージェント並行実行 | SP-7 並行タスク実行、アドホック協議での複数ロール同時検討 | 独立タスクの並行処理、マルチロール視点の同時収集 |
| バックグラウンドタスク | コーディングエージェントのテスト・ビルド実行 | テスト実行を非同期で行い、実装作業と並行させる |
| 対話型選択肢提示 | SP-6 Guided Prompt の実装 | オペレーターへの問いかけを選択肢形式で提示 |
| タスク進捗表示 | SP-7 並行タスク実行時の進捗可視化 | 並行実行中のタスク状態をリアルタイムで表示 |
| Web 検索・取得 | バグ修正手順のステップ3（公式ドキュメント確認） | 未知の問題に当たった場合の裏取り |

**メンテナンスポリシー:**
- Claude Code の新機能リリース時に、方法論のどのタッチポイントで活用できるかを評価し追記する
- 方法論ドキュメント（`.ai-native/methodology/`）には機能固有の記述を入れない。方法論は「何を」「なぜ」を定義し、本セクションが「どのように」を定義する
