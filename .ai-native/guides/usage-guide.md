# AIネイティブ開発 ドキュメント活用ガイド

> このガイドは、AIネイティブ開発手法のドキュメントセットを**人間が理解し、実際のプロジェクトで活用する**ための手引きです。

---

## この仕組みで何ができるのか

AIネイティブ開発は、AIを「便利なツール」としてではなく、**開発チームの構成員**として構造的に組み込む手法です。

この仕組みを導入すると:

- **あなた（オペレーター）は意思決定に専念**できる。実装・レビュー・テストはAIチームが実行する
- **5つの独立したAIロール**が相互に牽制し合うことで、単一AIに任せるよりも高い品質を構造的に保証する
- **9つのフェーズとゲート条件**により、「速く作ったが間違ったものだった」を防ぐ
- **壁打ちナビゲーション**により、課題発見から実装まで体系的に進められる

## 5つのAIロールとは

| ロール | 一言でいうと | あなたとの関係 |
|--------|------------|--------------|
| **PM・スクラムマスター** | 進捗を前に進める | 進捗レポートを受け取り、方針を判断する |
| **コーディングエージェント** | コードを書く | 実装結果を確認し、方向性を指示する |
| **コードレビュアー** | コード品質を止める | レビューサマリーを読み、承認/差し戻しを判断する |
| **システム監査官** | 安全性を止める | 監査サマリーを読み、リリース可否を判断する |
| **ユーザー・運用サポート** | ユーザーの代弁者 | テスト結果・UX評価を読み、改善を判断する |

**なぜ分けるのか？** 1つのAIにすべてを任せると、自分の実装を自分でレビューする構造になり、問題を見落としやすくなります。5ロールに分けることで、独立した視点からの相互チェックが機能します。

## セットアップ方法

### 方法1: Claude Code（CLAUDE.md に配置）

プロジェクトのルートに CLAUDE.md を作成し、必要なドキュメントを参照させます。

```
your-project/
├── CLAUDE.md              ← AIへの指示を記載
├── docs/
│   ├── core-principles.md ← コピーして配置
│   ├── phase-definitions.md
│   ├── review-standards.md
│   └── roles/
│       └── （必要なロールのプロンプトを配置）
├── src/
└── ...
```

**CLAUDE.md の記載例:**

```markdown
# プロジェクトルール

## AIネイティブ開発手法
docs/ 配下のドキュメントに従って開発を進めてください。

## 現在のフェーズ
Phase 5（基本設計＋プロトタイプ開発）

## 技術スタック
- フロントエンド: Nuxt3 / Vue3 / TypeScript
- バックエンド: Firebase Functions
- データベース: Firestore
- インフラ: Firebase Hosting
```

### 方法2: Claude Projects（プロジェクト知識に追加）

1. Claude.ai → Projects → 対象プロジェクトを開く
2. 「Project knowledge」に必要なドキュメントをアップロード
3. System prompt にロールを指定

**System prompt の例:**

```
あなたはコードレビュアーです。
アップロードされた core-principles.md と review-standards.md と code-reviewer.md に従って、
提出されたコードを7つのレビュー視点でレビューしてください。
```

### 方法3: Google AI Studio / ChatGPT GPTs

1. 新しいGem / GPT を作成
2. Instructions にロールプロンプトの内容を貼り付け
3. 共通ドキュメントの内容もInstructionsに含めるか、ファイルとしてアップロード

## 利用シーン別ガイド

### シーン1: 新しいプロジェクトをゼロから始める

**フェーズの進め方:**

| フェーズ | やること | 必要なドキュメント | 進捗管理 |
|---------|---------|-----------------|---------|
| Phase 0-4 | AIと壁打ちしながら課題発見→要件定義→技術選定 | core-principles + phase-definitions + navigator | ナビゲーターのセッションサマリーを元にオペレーターが progress-management.md に記録（任意） |
| Phase 5 | 設計とプロトタイプを同時に進める | + coding-agent | 同上 |
| Phase 6 | プロトタイプで関係者からフィードバックを集める | + user-ops-support | 同上 |
| Phase 7 | MVP構築。全ロールを稼働させる | 全5ロール + progress-management | PM・スクラムマスターが progress-management.md で管理 |
| Phase 8 | フルスケール実装 | 全5ロール + progress-management | 同上 |

**始め方:** AIに以下のように話しかけてください。

> 「新規プロジェクトを始めます。Phase 0からスタートしてください。」

### シーン2: 既存コードのレビューだけしたい

**必要なドキュメント:** core-principles + review-standards + code-reviewer

**始め方:**

> 「以下のコードを7つのレビュー視点でレビューしてください。」

### シーン3: セキュリティ監査だけしたい

**必要なドキュメント:** core-principles + review-standards + system-auditor

**始め方:**

> 「以下のコードの安全性・安定性・可用性を監査してください。データ機密度は高（個人情報を含む）です。」

### シーン4: 設計について壁打ちしたい

**必要なドキュメント:** core-principles + phase-definitions

**始め方:**

> 「この設計について壁打ちしたいです。現在Phase 5で、I/F設計のレビューをお願いします。」

### シーン5: 既存システムに機能を追加したい（Medium スコープ）

既存の設計・要件がある状態で中規模の機能を追加する場合。Phase 3 から開始する圧縮フロー。

**必要なドキュメント:** core-principles + phase-definitions + coding-agent + code-reviewer + system-auditor

**始め方:**

> 「既存システムに機能を追加します。Medium スコープで Phase 3 から開始してください。既存の要件定義は outputs/phase3/ にあります。」

### シーン6: バグを修正したい（Minimal スコープ）

既存の設計・アーキテクチャを変更しない小規模な修正。

**必要なドキュメント:** core-principles + review-standards + coding-agent（実装）、その後 code-reviewer + system-auditor（レビュー）

**始め方:**

> 「バグを修正します。Minimal スコープです。[バグの内容を説明]」

### シーン7: 本番障害に緊急対応したい（EMERGENCY_PATH）

本番環境の重大障害やセキュリティ脆弱性への緊急対応。

**必要なドキュメント:** core-principles + review-standards + coding-agent + code-reviewer + system-auditor

**始め方:**

> 「緊急対応パスを適用します。[障害の内容と重大度を説明]」

## 期待できる効果

### 品質面
- **二重ゲートによる品質保証:** コードレビュアー（品質）とシステム監査官（安全性）の2段階チェック
- **7つのレビュー視点の網羅:** データ設計、インターフェース設計、冗長性排除、変更耐性、エラーハンドリング、パフォーマンス（UI体感）、可読性
- **見落としの構造的な排除:** 異なる視点を持つ独立ロールが相互に補完

### 速度面
- **オペレーターは判断に専念:** 実装・レビュー・テストの作業負荷をAIに移譲
- **並行作業:** 5ロールが独立して動くため、レビューとテストを並行実行可能
- **ゲート条件による手戻り防止:** Phase 0-2で課題をしっかり掘り下げることで、後半の大きな手戻りを予防

### 追跡可能性
- **意思決定ログ:** いつ・誰が・なぜ判断したかが進捗管理テンプレートに蓄積され、経緯を追跡可能
- **フェーズ遷移履歴:** ゲート判定の結果と差し戻し理由が記録され、プロジェクトの経緯が第三者にも理解可能

### 再現性
- **判断基準の明文化:** 属人的だった判断基準がドキュメントとして共有可能
- **AIの一貫性:** 同じドキュメントを読ませれば、同じ基準でレビュー・監査を実行

## カスタマイズ

これらのドキュメントはベースラインです。プロジェクトごとに以下をカスタマイズしてください。

### レビュー基準の追加

```markdown
# 案件固有のレビュー追加基準

## 本案件の技術スタック
- Nuxt3 / Vue3 / TypeScript
- Firebase Firestore

## 追加レビュー基準
- Firestore のセキュリティルールが適切か
- Nuxt3 のauto-importに頼りすぎていないか
```

### フェーズのゲート条件追加

```markdown
# 案件固有のゲート条件

## Phase 0-2 追加ゲート条件
- クライアントの既存システムとの連携要件が明確化されているか

## Phase 7 MVP追加判定基準
- i18n対応が完了しているか（対象言語: 日本語・英語）
```

## よくある質問

**Q: 5ロール全部を同時に使わないといけないの？**
A: いいえ。フェーズに応じて必要なロールだけ使えます。Phase 0-4は共通ドキュメントだけで十分です。全ロール稼働はPhase 7以降が目安です。

**Q: 1つのAIに全ロールを兼務させてもいい？**
A: 推奨しません。自分の実装を自分でレビューする構造は、問題の見落としを招きます。最低限、実装とレビューは別のAIセッションで行ってください。

**Q: ドキュメントの量が多くてAIのコンテキストに入りきらない場合は？**
A: ロール別のドキュメントセット（INDEX.md に定義）に従って、必要最小限のドキュメントだけを読み込ませてください。全部を一度に読ませる必要はありません。
