# レビュー出力テンプレート

品質ゲート（コードレビュアー）と安全ゲート（システム監査官）の出力形式を統合し、
オペレーターが最終判断を下すまでの一連の出力テンプレートを定義する。

> **進捗管理との連携:** 本テンプレートによるレビュー・監査結果は、PM・スクラムマスターにより進捗管理テンプレート（progress-management.md §6 レビュー・監査サマリー）に蓄積される。

---

## 1. コードレビュアー出力テンプレート（品質ゲート）

コードレビュアーはこのテンプレートに従って出力する。
7つのレビュー視点すべてに対して判定・根拠を記載すること。1つでも欠落させてはならない。

```markdown
# コードレビューサマリー（品質ゲート）

## レビュー対象
- **対象機能:** （機能名を記載）
- **対象ファイル/モジュール:** （ファイルパスやモジュール名を列挙）
- **レビュー実施日:** YYYY-MM-DD
- **レビュー実施者:** コードレビュアー（AIセッションID等があれば記載）

## コンテキスト
- **技術スタック:** （言語、フレームワーク、DB種別、インフラ構成）
- **データ機密度:** 高（個人情報・決済情報あり）/ 中（社内データ）/ 低（その他）
- **動作環境:** （同居システム、リソース上限等）
- **フェーズ:** Phase 5（プロトタイプ）/ Phase 7（MVP）/ Phase 8（フルスケール）
- **パフォーマンス基準:** 初期表示200ms以下 / 検索100ms以下（または案件固有の基準値）

## 実行順序
データ層 → インターフェース層 → コード層 → 非機能層 の順で実施。
上位層に問題がある場合、下位層のレビューは保留とした。
```

## 2. システム監査官出力テンプレート（安全ゲート）

システム監査官はこのテンプレートに従って出力する。
安全性・安定性・可用性・致命的パターンの4領域すべてに対して判定を記載すること。

```markdown
# システム監査サマリー（安全ゲート）

## 監査対象
- **対象機能:** （機能名を記載）
- **対象ファイル/モジュール:** （ファイルパスやモジュール名を列挙）
- **監査実施日:** YYYY-MM-DD
- **監査実施者:** システム監査官（AIセッションID等があれば記載）

## 監査コンテキスト
- **データ機密度:** 高 / 中 / 低
- **動作環境:** （クラウド/オンプレ、コンテナ設定等）
- **同居システム:** （同じ環境で稼働するアプリケーション一覧）
- **リソース上限:** CPU: X / Memory: Xm / Storage: XGB
- **同時接続ユーザー数（想定最大）:** X人
- **フェーズ:** Phase 7（MVP）/ Phase 8（フルスケール）

---

## 監査結果

### 安全性（Security）
- **判定:** PASS / FAIL
- **監査深度:** 最深度（個人情報・決済情報あり）/ 深度（社内データ）/ 標準
- **確認事項:**
  - データ保管: （暗号化状況、アクセス範囲）
  - アクセス経路: （認証・認可、API保護）
  - フロントエンド: （クレデンシャル露出、DOM/ネットワーク露出）
- **指摘事項:** （あれば具体的に記載。なければ「検出なし」）

### 安定性（Stability）
- **判定:** PASS / FAIL
- **確認事項:**
  - 障害時の振る舞い: （フェイルセーフ設計の有無）
  - リカバリ手順: （有無と妥当性）
  - データ一貫性: （トランザクション設計）
  - エラーで停止する箇所: （有無）
  - クリティカルログ: （取得設計の有無）
- **指摘事項:**

### 可用性（Availability）
- **判定:** PASS / FAIL
- **確認事項:**
  - 単一障害点（SPOF）: （有無）
  - スケーラビリティ: （設計状況）
  - 同居アプリへの影響: （CPU/メモリの競合リスク）
  - キャパシティプランニング: （想定負荷との整合）
- **指摘事項:**

### 致命的パターン検出
- **メモリリーク:** 検出なし / 検出あり → （詳細: ライブラリ名、コード箇所）
- **無限ループ:** 検出なし / 検出あり → （詳細: 条件、コード箇所）
- **デッドロック:** 検出なし / 検出あり → （詳細: ロック対象、コード箇所）
- **ストレージ永久増加:** 検出なし / 検出あり → （詳細: ファイル出力箇所、ローテート設定）

---

## 指摘事項一覧

| # | 分類 | 重大度 | 内容 | リスク | 対応案 | 該当箇所 |
|---|------|--------|------|--------|--------|---------|
| 1 | Security | CRITICAL | ... | ... | ... | ... |
| 2 | Stability | WARNING | ... | ... | ... | ... |

**重大度の定義:**
- **CRITICAL:** 即対応が必要。リリース絶対不可。全ロールに最優先で通知
- **WARNING:** 対応を推奨。リリース前に解決すべき
- **INFO:** 認識しておくべきリスク。運用で対応可能

## 総合判定
- **安全ゲート:** PASS / FAIL
- **リリース可否:** リリースOK / リリース不可
- **CRITICAL指摘数:** X件
- **WARNING指摘数:** X件
- **判定根拠:** （PASS/FAILの理由を具体的に記載）
- **次のアクション:** オペレーター判断へ引き渡し / コーディングエージェントへ差し戻し
```

## 3. オペレーター判断テンプレート

品質ゲートと安全ゲートの両方のサマリーを受け取った後、オペレーターが最終判断を記録するテンプレート。

```markdown
# オペレーター最終判断

## 対象
- **対象機能:** （機能名）
- **判断日:** YYYY-MM-DD

## 入力サマリー

### 品質ゲート（コードレビュアー）
- **結果:** PASS / FAIL
- **CRITICAL:** X件 / **WARNING:** X件 / **INFO:** X件

### 安全ゲート（システム監査官）
- **結果:** PASS / FAIL
- **CRITICAL:** X件 / **WARNING:** X件 / **INFO:** X件

## オペレーター検証チェックリスト

- [ ] 7視点すべてがサマリーに含まれている（データ設計・I/F設計・冗長性・変更耐性・エラーハンドリング・パフォーマンス・可読性）
- [ ] 各視点の判定根拠が具体的である（「問題なし」だけでなく、何を確認したかが明示されている）
- [ ] 指摘事項の重大度判定が妥当である（データ設計・I/F設計の問題が軽微扱いされていない）
- [ ] ドメイン知識から見て見落としがない（業界固有の制約、レガシーシステムとの互換性等）
- [ ] 安全ゲートの4領域（安全性・安定性・可用性・致命的パターン）がサマリーに含まれている
- [ ] 安全ゲートの致命的パターン検出結果を確認した

## 判断

- **最終判断:** 承認 / 差し戻し（品質ゲート修正） / 差し戻し（安全ゲート修正） / 差し戻し（両方修正）
- **判断根拠:** （なぜ承認/差し戻しと判断したか）
- **追加指示:** （差し戻しの場合、修正の優先順位や具体的な指示）
- **許容事項:** （WARNING/INFOで今回は許容するものがあれば、その理由）
- **技術的負債登録:** （許容した WARNING/INFO を技術的負債レジストリに登録する。progress-management.md §7 参照）

| # | 許容した指摘 | 重大度 | 許容理由 | 計画解消時期 |
|---|------------|--------|---------|------------|
| 1 | （指摘の概要） | WARNING/INFO | （なぜ今回は許容するか） | （いつ対応するか） |
```

> **記録:** この判断の後、PM・スクラムマスターが進捗管理テンプレート（progress-management.md §6 レビュー・監査サマリー）に結果を記録し、許容した指摘を §7 技術的負債レジストリに登録する。

---

## 出力テンプレートの使い方

### レビューフロー全体

```
1. コーディングエージェント → 実装完了報告（coding-agent.md の OUTPUT_FORMAT）
2. コードレビュアー     → 本テンプレート §1 で品質ゲートサマリーを出力
3. システム監査官       → 本テンプレート §2 で安全ゲートサマリーを出力
4. オペレーター         → 本テンプレート §3 で最終判断を記録
```

### インクリメンタルレビュー（Phase 5-8）

@ref:core-principles#SP-8 に基づき、実装タスク完了ごとにインクリメンタルレビューが実施される。
インクリメンタルレビューは §4 の軽量テンプレートを使用する。§1-§3 のフルテンプレートはゲートレビュー時に使用する。

### テンプレートのカスタマイズ

案件固有の追加チェック項目がある場合、各テンプレートの「確認事項」に項目を追加して使用する。
ただし、7つのレビュー視点と4つの監査領域を削減してはならない。

---

## 4. インクリメンタルレビューテンプレート（SP-8）

@ref:core-principles#SP-8 に基づく実装タスク完了ごとのレビュー/監査テンプレート。
ゲートレビュー（§1-§3）とは異なり、変更スコープに限定した軽量テンプレートを使用する。
判定値は CLEAR / ISSUE（ゲートの PASS / FAIL と区別）。

### 4.1 インクリメンタル品質チェックテンプレート

コードレビュアーが実装タスク完了ごとに使用する軽量テンプレート。

```markdown
# インクリメンタル品質チェック

## チェック対象
- **対象タスク:** （タスク名/概要）
- **変更ファイル:** （変更されたファイルパスを列挙）
- **チェック実施日:** YYYY-MM-DD
- **イテレーション:** 初回 / 第N回（前回 ISSUE からの再チェック）

## スコープ宣言
- **変更ファイル:** （git diff で特定されるファイル一覧）
- **直接影響I/F:** （変更された API/型を利用するファイル。該当なしの場合は「なし」）
- **データモデル影響:** （スキーマ変更時の依存コードパス。該当なしの場合は「なし」）
- **スコープ外の除外理由:** （レビュー対象外とした範囲があれば理由を記載）

## 視点別チェック結果

| # | 視点 | 判定 | 確認内容 | 根拠アーティファクト |
|---|------|------|---------|-------------------|
| 1 | データ設計 | CLEAR/ISSUE/N/A | （確認した内容 or N/A の理由） | （ファイル:行番号） |
| 2 | I/F 設計 | CLEAR/ISSUE/N/A | ... | ... |
| 3 | 冗長性排除 | CLEAR/ISSUE | ... | ... |
| 4 | 変更耐性 | CLEAR/ISSUE | ... | ... |
| 5 | エラーハンドリング（エラーコード付与含む） | CLEAR/ISSUE | ... | ... |
| 6 | パフォーマンス | CLEAR/ISSUE/N/A | ... | ... |
| 7 | 可読性 | CLEAR/ISSUE | ... | ... |

## 指摘事項（ISSUE がある場合）

| # | 視点 | 重大度 | 内容 | 理由 | 対応案 | コード箇所 |
|---|------|--------|------|------|--------|-----------|
| 1 | ... | CRITICAL/WARNING/INFO | ... | ... | ... | ファイル:行番号 |

## 総合判定
- **インクリメンタル品質チェック:** CLEAR / ISSUE
- **次のアクション:** システム監査官へ引き渡し（CLEAR時）/ コーディングエージェントへ差し戻し（ISSUE時）
```

### 4.3 トリビアルスコープ確認テンプレート

コーディングエージェントが「トリビアルスコープ」を宣言した場合に、コードレビュアーが使用する。

```markdown
# トリビアルスコープ確認

- **対象タスク:** （タスク名/概要）
- **変更ファイル:** （変更されたファイルパスを列挙）
- **コーディングエージェントの宣言根拠:** （ランタイム非影響の理由）
- **確認結果:** 確認（トリビアルスコープとして承認） / 却下（通常のインクリメンタルレビューを実施）
- **却下理由:** （却下の場合のみ記載）
```
