## §3 意思決定ログ

プロジェクトにおける重要な意思決定を時系列で記録する。
「なぜその判断をしたのか」を残すことで、後から経緯を追跡可能にする。

### 記録ルール

- **何を記録するか:** フェーズの進退判断、技術選定、要件の取捨選択、設計方針の決定、差し戻し判断、フィードバックへの対応方針
- **誰が記録するか:** Phase 0-6はオペレーター（ナビゲーターのセッションサマリーを元に）。Phase 7-8はPM・スクラムマスター
- **いつ記録するか:** 意思決定が行われた直後。後回しにしない

### 意思決定ログ

| # | 日付 | フェーズ | カテゴリ | 意思決定の内容 | 判断理由 | 判断者 | 影響範囲 |
|---|------|---------|---------|--------------|---------|--------|---------|
| 1 | YYYY-MM-DD | Phase X | （下記参照） | 何を決めたか | なぜそう判断したか | 誰が判断したか | 何に影響するか |

**カテゴリ一覧:**

| カテゴリ | 説明 | 記録タイミング |
|---------|------|--------------|
| 課題定義 | 顧客の真の課題の特定・修正 | Phase 0-1 |
| ユースケース | ユースケースの追加・変更・削除 | Phase 2 |
| 要件決定 | 機能要件・非機能要件の確定・変更 | Phase 3 |
| 技術選定 | 技術スタックの選定・変更 | Phase 4 |
| 設計方針 | I/F設計・データ設計・API設計の決定 | Phase 5 |
| フィードバック対応 | フィードバックへの対応方針 | Phase 6 |
| 実装判断 | 実装スコープ・優先度の決定 | Phase 7-8 |
| 品質判断 | レビュー指摘への対応方針 | Phase 7-8 |
| 安全判断 | 監査指摘への対応方針 | Phase 7-8 |
| 差し戻し | フェーズ差し戻しの判断 | 全フェーズ |

Phase 7-8のMVP構築・フルスケール実装で使用。タスク単位の進捗を管理する。

### 現在のイテレーション

**イテレーション:** #X　**期間:** YYYY-MM-DD 〜 YYYY-MM-DD

**イテレーション目標:** （このイテレーションで達成すること）

| # | タスク | 担当ロール | 優先度 | 状態 | 完了日 | 備考 |
|---|--------|-----------|--------|------|--------|------|
| 1 | | コーディングエージェント | 高/中/低 | 未着手/進行中/レビュー中/完了 | | |
| 2 | | コードレビュアー | 高/中/低 | 未着手/進行中/完了 | | |

**状態の定義:**
- **未着手:** まだ作業が開始されていない
- **進行中:** 作業中
- **レビュー中:** コードレビュアーまたはシステム監査官の確認待ち（インクリメンタルレビュー（SP-8）およびゲートレビューの両方を含む）
- **差し戻し:** レビュー/監査で差し戻された（要修正）
- **完了:** 作業完了、品質/安全ゲートも通過済み

> **インクリメンタルレビュー記録について:** SP-8 インクリメンタルレビューの結果（CLEAR/ISSUE）は §6 レビュー・監査サマリーに蓄積する。ゲートレビュー時の入力として活用されるため、漏れなく記録すること。

## §7 技術的負債レジストリ

レビュー・監査で検出されたが許容された WARNING/INFO 指摘を経時追跡する。蓄積した妥協が構造的問題に成長することを防ぐ。

### 登録ルール

- **登録タイミング:** オペレーター判断テンプレート（`review-output-template.md` §3）で WARNING/INFO を許容した時点で即登録
- **登録者:** PM・スクラムマスター
- **レビュー:** イテレーション境界で PM がレジストリをレビューし、蓄積傾向をオペレーターに報告する
- **閾値:** 未解消の WARNING が10件を超えた場合、技術的負債の解消を目的とした強制レビューをトリガーする

### 技術的負債レジストリ

| # | 登録日 | 発生源 | 重大度 | 内容 | 許容理由 | 推定影響 | 計画解消時期 | 解消日 | 状態 |
|---|--------|--------|--------|------|---------|---------|------------|--------|------|
| 1 | YYYY-MM-DD | レビュー/監査（日付） | WARNING/INFO | （指摘の概要） | （なぜ許容したか） | （放置した場合の影響） | Phase X / イテレーション #Y | | 未解消/解消済 |

### 技術的負債サマリー

| 項目 | 値 |
|------|-----|
| **総登録数** | X件 |
| **未解消数** | Y件 |

## 使い方ガイド

### 更新タイミング

| イベント | 更新セクション |
|---------|--------------|
| フェーズ移行（ゲートPASS） | §1 ダッシュボード + §2 遷移履歴 |
| 意思決定が行われた | §3 意思決定ログ |
| イテレーション開始/終了（Phase 7-8） | §4 スプリント進捗 |
| ブロッカー検知/解消 | §5 ブロッカー管理 |
| リスク検知/状態変化 | §5 リスク管理 |
| レビュー/監査完了 | §6 レビュー・監査サマリー |
| 技術的負債の登録/解消 | §7 技術的負債レジストリ |
| オペレーター委任の開始/終了 | §8 オペレーター/代理の管理 |
| 差し戻し発生 | §2 遷移履歴 + §3 意思決定ログ |

### 記録の粒度

- **ダッシュボード（§1）:** 常に最新の状態を反映する「スナップショット」。過去の情報は上書きしてよい
- **遷移履歴（§2）:** 追記のみ。過去の記録を変更・削除しない
- **意思決定ログ（§3）:** 追記のみ。過去の記録を変更・削除しない
- **スプリント進捗（§4）:** 現在のイテレーションは随時更新。完了したイテレーションは履歴に移動
- **ブロッカー/リスク（§5）:** 状態を更新する。解消済みは解消日を記入
- **レビュー・監査（§6）:** 追記のみ
- **技術的負債（§7）:** 追記のみ。解消時は解消日と状態を更新
- **オペレーター/代理（§8）:** 委任開始時に追記。期間終了時に更新
