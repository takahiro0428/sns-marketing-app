# Review Standards

レビューは上位の層から順に確認する。データ層に問題があればコード層の議論は後回しにする。
土台が腐っていれば上に何を積んでも意味がない。

すべてのレビュー判断は @ref:core-principles#TOP_LEVEL_PRINCIPLE「ユーザーの意図を完遂させる」に帰結する。

## REVIEW_ARCHITECTURE（全体構造）

```
[最上位原則] ユーザーの意図を完遂させる
│
├─ [データ層] ── 変更耐性の土台
│   ├─ 正規化
│   ├─ キー設計
│   └─ マスタ設計
│
├─ [インターフェース層] ── システムの柔軟性の要
│   ├─ API責務の分離（癒着排除）
│   ├─ I/Fファースト設計
│   └─ データフロー全体の整合性
│
├─ [コード層] ── 実装品質
│   ├─ 冗長性排除
│   ├─ 変更耐性（分岐の将来性判断）
│   ├─ 可読性
│   └─ エラーハンドリング
│
├─ [非機能層] ── システムの信頼性
│   ├─ パフォーマンス
│   ├─ セキュリティ
│   └─ リソースコスト
│
└─ [横断原則]
    ├─ 汎用化はユーザー価値に貢献する場合のみ
    ├─ 用途のない抽象化は入れない
    └─ 他者が読んで意図がわかるか
```

> **ゲート別の担当範囲:**
> - **品質ゲート（コードレビュアー）:** データ層・インターフェース層・コード層の品質、およびUI体感パフォーマンス
> - **安全ゲート（システム監査官）:** セキュリティ・安定性・可用性・リソースコスト・致命的パターン検出

## LAYER_1: データ層（変更耐性の土台）

### 1.1 正規化

```
[データは正規化されているか？]
├─ YES → OK
└─ NO → 非正規化の根拠があるか？
    ├─ read/writeコストバランスの明確な理由がある → 許容（根拠を記録）
    └─ 根拠がない → 正規化を指示
```

- 基本方針はデータを正規化してきれいに揃えること
- 非正規化は例外措置。要件・仕様に基づくread/writeコストバランスの根拠が必要
- 「パフォーマンスのため」という漠然とした理由は不可。具体的な計測値や想定アクセスパターンを求める

### 1.2 キー設計

```
[主キーの設計は適切か？]
├─ 意味を持たないユニークID（UUID等）がPKか？
│   ├─ YES → そのIDがrelationに使用されているか確認
│   └─ NO → 業務的な意味を持つキーがPK
│       → 変更リスクを指摘。アジャイル開発ではPKの変更は致命的
│
├─ 複合キーで強い制約を作っていないか？
│   ├─ 複合キーがある → 変更耐性リスクを警告
│   └─ ない → OK
```

- アジャイル前提では、テーブルごとに意味を持たないユニークIDをPKとしrelationにはそのIDを使用
- 複合キーによる強い制約はDB変更に弱い。物事が確定するまでは避ける

### 1.3 マスタ設計

```
[マスタデータの設計は適切か？]
├─ selectやautocomplete等のUI要素に使うデータ → マスタ化されているか確認
├─ 静的マスタか動的マスタか判断されているか？
│   ├─ 静的（コード値等） → 変更頻度が極めて低いことが確認されているか
│   └─ 動的（ユーザーが追加・編集可能） → CRUD機能の設計があるか
```

## LAYER_2: インターフェース層（システムの柔軟性の要）

### 2.1 API責務の分離（癒着排除）

```
[APIに癒着がないか？]
├─ パターンA: 責務の混同
│   └─ 一覧取得と詳細取得が分離されていないケース
│       → クライアント側にグループ化・集約の責務を押し付けていないか？
│       → 分離すべき。1つのAPIは1つの責務
│
├─ パターンB: レスポンスの肥大化
│   └─ 1つのエンドポイントに関連データを全部載せているケース
│       → 別リソースとして切り出す
```

- 1つのAPIが1つの責務を超えていないか
- クライアントに集約・加工の責務を押し付けていないか
- API定義を見たとき、使い方が直感的にわかる単位で切られているか

### 2.2 I/Fファースト設計 6視点

| # | 視点 | 確認内容 |
|---|------|---------|
| 1 | 技術スタック制約 | RDB/ドキュメントDB/物理ファイルのどれが適切か |
| 2 | ユースケース | 壁打ちログからのユーザー利用シーン想定 |
| 3 | ユーザビリティ | UIの最適な入出力表現に適合するデータ形式か（USABILITY_STANDARDS §設計時ユーザビリティ U-1〜U-4 で詳細確認） |
| 4 | データ設計上の都合 | マスタの静的/動的、select/autocomplete等のUI要素との連動 |
| 5 | 型の継承関係 | class/interface/typeの継承・実装関係に矛盾がないか |
| 6 | データフロー整合性 | 起点のI/Fから派生する全I/Fが一気通貫で矛盾しないか |

### 2.3 データフロー全体の整合性

```
[I/F間のデータフローに矛盾がないか？]
├─ 起点のI/Fから終点まで、データの型・構造が一貫しているか
├─ 途中で不要な変換・マッピングが発生していないか
├─ フロー上の各I/Fが6視点を満たしているか
└─ 複数データストア間の SoT が宣言され、同期パスが完全か
    ├─ 各データの SoT はどのシステムか明示されているか
    ├─ SoT → キャッシュの書込順序が守られているか
    ├─ SoT からの回復パス（手動同期等）が存在するか
    └─ データフロー検証レポートのリスク分析を参照
```

## USABILITY_STANDARDS（ユーザビリティ基準）

ユーザビリティは I/F設計（Phase 5）からUX検証（Phase 6-8）まで横断的に適用される。
本セクションは各フェーズ・ロールが参照する統一基準を定義する。

### 設計時ユーザビリティ（Phase 5 I/F設計で適用）

I/Fファースト設計6視点の第3項「ユーザビリティ」を以下の観点で具体化する:

| # | 観点 | 確認内容 |
|---|------|---------|
| U-1 | 入力の最適化 | ユーザーの入力負荷を最小化するデータ形式か（選択式 vs 自由入力、デフォルト値、入力補助） |
| U-2 | 出力の直感性 | 表示データがユーザーの業務文脈で即座に理解できる形式か（並び順、グルーピング、ラベル） |
| U-3 | 操作フローの自然さ | ユーザーの業務フローに沿った画面遷移・操作順序か（不要な戻り操作、迂回が発生しないか） |
| U-4 | エラー時の誘導 | エラー状態からユーザーが自力で復帰できるUI設計か（エラーメッセージ、次のアクションの明示） |
| U-5 | アクセシビリティ配慮 | キーボード操作、スクリーンリーダー対応、色のコントラスト比等、多様なユーザーが利用可能なUI設計か。適用レベルはプロジェクト非機能要件に従う |

### 検証時ユーザビリティ（Phase 6-8 UX検証で適用）

ユーザー・運用サポートのUX検証で適用する統一基準。UX検証レポートの5次元はこの定義に準拠する:

| # | 次元 | 判定基準 |
|---|------|---------|
| UX-1 | 業務フロー整合性 | 実際の業務手順と画面操作の順序が一致しているか |
| UX-2 | 画面遷移の自然さ | ユーザーが「次に何をすればいいか」を迷わないか |
| UX-3 | データ表示の適切さ | 表示の粒度・順序・形式が業務判断に最適か |
| UX-4 | エラーUX | エラー発生時にユーザーが状況を理解し、次のアクションを取れるか |
| UX-5 | 操作迷いリスク | ユーザーが操作を躊躇する・間違えるポイントがないか |
| UX-6 | アクセシビリティ | 多様なユーザー（視覚・聴覚・運動機能に制約のあるユーザー等）が利用可能か。適用レベルはプロジェクトの非機能要件で決定する |

### 設計時と検証時の対応関係

設計時の観点（U-1〜U-4）で適切に設計されたI/Fが、検証時の次元（UX-1〜UX-5）で妥当性を確認される。対応関係は以下の通り:

| 設計時（I/F設計） | → | 検証時（UX検証） |
|------------------|---|-----------------|
| U-1 入力の最適化 | → | UX-1 業務フロー整合性、UX-5 操作迷いリスク |
| U-2 出力の直感性 | → | UX-3 データ表示の適切さ、UX-5 操作迷いリスク |
| U-3 操作フローの自然さ | → | UX-2 画面遷移の自然さ、UX-5 操作迷いリスク |
| U-4 エラー時の誘導 | → | UX-4 エラーUX |
| U-5 アクセシビリティ配慮 | → | UX-6 アクセシビリティ |

## LAYER_3: コード層（実装品質）

### 3.1 冗長性排除

```
[似た処理が複数箇所にある]
├─ 引数追加で統合可能か？
│   ├─ YES → 統合後の関数が長くなりすぎないか？
│   │   ├─ YES → 共通部分だけ基礎関数に切り出し、各処理はラッパーで分離
│   │   └─ NO → 統合した関数の責務は単一か？
│   │       ├─ YES → 統合する
│   │       └─ NO → 責務ごとに分けて、共通部分だけ基礎関数に抽出
│   └─ NO → 処理の意味的な類似度は高いか？
│       ├─ YES → 抽象化の設計を見直す
│       └─ NO → 別関数のまま維持（重複に見えるが責務が異なる）
```

- 冗長の判定は行数ではなく**責務の重複**で判断
- 統合はメンテナンス性の向上が目的
- 意味的な境界を尊重する

### 3.2 変更耐性（分岐の将来性判断）

```
[分岐処理がある場合]
├─ 分岐の種類は増える可能性があるか？
│   ├─ 確認済み・増えない → 最短・高パフォーマンスな分岐でOK
│   ├─ 未確認 → 即確認させる。確認なく実装を進めるのは不可
│   └─ 増える or 種類未確定
│       → 固定値を避け、enum等の定義で動かせる設計にする
│
[汎用化の判断]
├─ 開発効率性・システム拡張性・可用性への貢献があるか？
│   ├─ YES → パフォーマンスの軽微な影響は許容し積極的に汎用化
│   └─ NO → 用途のない汎用性は入れない
```

### 3.3 可読性

```
[可読性判断]
├─ 定義の明確さ
│   ├─ マジックナンバーが使われていないか → enum等で定義
│   ├─ 型定義が適切に整備されているか
│   └─ サマリー・コメントが最低限あるか
│
├─ 構造の意図
│   ├─ フォルダ構成に設計意図があるか（レイヤーが意識されているか）
│   ├─ helpers/utils/extensionsが意味なく乱立していないか
│   └─ クラスの分割単位に責務の境界があるか
│
└─ 粒度の適切さ
    ├─ 1メソッドが異常に長くないか
    ├─ 1クラスが肥大化していないか
    └─ ネスト・ジャンプが深くなっていないか
```

- 可読性の本質は「他者への配慮の有無」
- フォルダ構成は設計思想の表出
- 1メソッド・1クラスの肥大化は設計不足のサイン

### 3.4 エラーハンドリング

```
[ユーザーが意図した操作を完遂できたか？]
├─ YES → 正常終了
└─ NO → なぜ？
    ├─ 必要条件を満たしていない
    │   ├─ 理由をユーザーに明示する
    │   ├─ ユーザーのアクションで解決可能 → アクションを誘導
    │   └─ ユーザーでは解決不可能 → 下記へ
    │
    ├─ システムエラー
    │   ├─ エラー概要をユーザーに簡潔に通知（技術的詳細は不要）
    │   ├─ 自動復旧可能 → 自動復旧し結果を通知
    │   └─ 自動復旧不可 → 人的サポートへ誘導
    │
    └─ [共通] ログ設計
        ├─ クリティカルエラー → 無条件で取得
        ├─ ワーニング・軽微 → 必要時に取得できる設計
        └─ ログの粒度を切替可能な設計
```

- エラーで落ちて止まるのは論外
- 想定できるエラーはすべてキャッチ
- レビュー提出前にエラーハンドリング設計が完了していること
- **エラーコード管理:**
  - 想定されるエラーにはエラーコードを付与し、コードベースで一元管理されていること
  - エラーコードの命名規則はプロジェクトで定義すること（方法論では体系の存在のみを要求）
  - エラーコードからエラー内容・対処方法を特定できる構造であること

### 3.5 テスト設計

```
[テストが適切に設計・実装されているか？]
├─ テストが存在するか？
│   ├─ YES → テストの網羅性を確認
│   └─ NO → テストの作成を指示
│       ├─ 新規機能の場合 → ユニットテスト必須
│       └─ 既存機能の修正 → 変更箇所に対するテスト必須
│
├─ テストの網羅性は十分か？
│   ├─ 正常系の主要ユースケースがカバーされているか
│   ├─ 異常系（バリデーションエラー、境界値）がカバーされているか
│   └─ 結合に影響する変更の場合、インテグレーションテストがあるか
│
├─ テストの品質は適切か？
│   ├─ テストが実装の内部構造に過度に依存していないか（脆いテスト）
│   ├─ テスト名・構造からテスト対象の振る舞いが読み取れるか
│   └─ モック・スタブの使用が適切か（過度なモックで実質無意味になっていないか）
│
└─ 全テストスイートがパスしているか？
    ├─ YES → OK
    └─ NO → 失敗原因の特定と修正を指示
```

- テストの目的は「壊れたことの早期検知」であり、カバレッジ数値の達成ではない
- 既存テストが壊れている場合、意図的な振る舞い変更か、リグレッションかを判別する
- テストがない状態でのレビューPASSは原則不可。プロトタイプフェーズ（Phase 5）のみ例外とする

## LAYER_4: 非機能層（システムの信頼性）

### 4.1 パフォーマンス

最終判断基準は**UI体感**。ユーザーの体感はUIでしか起こらない。

| 項目 | 基準値 | 超過時の対応 |
|------|--------|-------------|
| 画面に変化がない時間 | 100ms以下を目指す | -- |
| 初期表示 | 200ms以下 | 非同期ロード+スケルトン等で体感補償 |
| 検索結果表示 | 100ms以下 | キュー管理+完了時プッシュ表示 |
| 上記を超える処理 | -- | 非同期処理+完了通知 |

- ユーザーにストレスを与えるシステムであってはならない
- すべての処理が数値基準を満たす必要はないが、体感上の停止感は排除する
- **クリティカルなパフォーマンス問題の扱い:** IQ-1 が許容する「軽微な低下」とは汎用化に伴うminorなトレードオフを指す。ユーザー体感に明確な悪影響を与えるクリティカルなパフォーマンス問題は「軽微」の範囲外であり、改修コストの大小にかかわらず対処を検討すること

> **ゲート間の分担:** UI体感のパフォーマンス（レスポンス時間、画面の停止感）は品質ゲート（コードレビュアー RP-6）で検証する。インフラレベルの負荷・キャパシティ・スケーラビリティは安全ゲート（システム監査官 AS-3）で検証する。

### 4.2 セキュリティ

```
[データの機密度で深度を決定]
├─ 最優先: 個人情報・決済情報 → 最深度レビュー
├─ 次点: 社内データ → 同等の視点で深くレビュー
└─ その他 → 標準レビュー

├─ データ保管
│   ├─ 保管先DBはどの環境から参照可能か
│   ├─ 秘匿データが適切に分離されているか
│   └─ 暗号化されているか
│
├─ アクセス経路
│   ├─ アクセス手順は制御されているか
│   ├─ どのシステムが参照しているか把握されているか
│   └─ API保護（トークン認証、危険なコントローラの露出、多層API構成、インバウンド制御）
│
└─ フロントエンド固有
    ├─ キー・クレデンシャルがコードや静的ファイルに露出していないか
    ├─ 秘匿情報がDOMやネットワークタブから見えていないか
    └─ 容易にスクレイピングされる状態になっていないか
```

### 4.3 リソースコスト

```
[リソースコスト判断]
├─ 環境全体の設計
│   ├─ 同居するアプリケーションは何か
│   ├─ 各アプリのCPU・メモリ占有率の上限設定は適切か
│   └─ 全アプリの最重処理が同時発生した場合の想定はあるか
│
├─ キャパシティプランニング
│   ├─ 同時接続ユーザー数の最大想定は妥当か
│   ├─ インフラのサイジングは決定されているか
│   └─ 瞬間的な負荷増に対する拡張は検討されているか
│
├─ 定時タスク・バッチ
│   └─ 実行時間帯の負荷を考慮した設計か
│
├─ ファイル・ストレージ
│   ├─ 動作環境内にファイル出力があるか
│   ├─ 永久に増え続けるロジックになっていないか
│   └─ ローテートが有効か
│
└─ 致命的パターンの検出（絶対に見逃さない）
    ├─ メモリリークの危険があるライブラリ・コンテナイメージ
    ├─ 無限ループの発生可能性
    └─ デッドロックの発生可能性
```

## REVIEW_OUTPUT_FORMAT（レビュー出力形式）

各指摘には以下を含めること:

| 項目 | 内容 |
|------|------|
| **視点** | どの視点での指摘か（冗長性、変更耐性、セキュリティ等） |
| **重大度** | CRITICAL（即修正）/ WARNING（修正推奨）/ INFO（改善提案） |
| **理由** | なぜ問題なのか（最上位原則との関連を明記） |
| **対応案** | 具体的な修正方針 |

品質ゲート・安全ゲート・オペレーター判断を統合した詳細な出力テンプレートは templates/review-output-template.md を参照。

## REVIEW_CONTEXT（レビュー実行前に提供すべき情報）

1. プロジェクトの技術スタック（言語、フレームワーク、DB種別、インフラ構成）
2. 扱うデータの機密度（個人情報・決済情報の有無）
3. 動作環境の制約（同居するシステム、リソース上限）
4. 現在のフェーズ（プロトタイプ / MVP / リリース前 / 運用中）
5. パフォーマンス基準（デフォルトは本ドキュメントの基準値を適用）

## REVIEW_EXECUTION_ORDER（実行順序）

1. データ層 → 2. インターフェース層 → 3. コード層 → 4. 非機能層

上位層に問題がある場合、下位層のレビューは保留し、上位層の修正を優先指示する。

## INCREMENTAL_REVIEW_SCOPE（インクリメンタルレビュースコープ）

@ref:core-principles#SP-8 に基づき、実装タスク完了ごとに変更範囲に対するレビュー/監査を実施する。
本セクションはインクリメンタルレビューのスコープ定義と視点適用ルールを定める。

### スコープ定義

インクリメンタルレビューの対象範囲は以下の3層で定義する:

| 層 | 対象 | 説明 |
|---|------|------|
| 1. 変更ファイル | 直接変更されたコード | git diff で特定される変更差分 |
| 2. 直接影響I/F | 変更された API/型を利用するファイル | 型定義の変更、API シグネチャの変更が影響する呼び出し元 |
| 3. データモデル影響 | スキーマ変更時の依存コードパス | データモデル変更がある場合のみ適用。マイグレーション・クエリ・バリデーションへの影響 |

レビュアー/監査官はスコープ境界を出力に明示し、除外した範囲とその理由を記載すること。

### フェーズ別の適用モード

| フェーズ | モード | 品質チェック（コードレビュアー） | 安全チェック（システム監査官） |
|---------|--------|-------------------------------|------------------------------|
| Phase 5（プロトタイプ） | 軽量デザインレビュー | RP-1, RP-2 を中心に。RP-3〜RP-7 は重大な問題のみ | AS-4（致命的パターン）のみ |
| Phase 6（フィードバック） | フィードバック反映確認 | RP-2（I/F変更の整合性）、RP-7（可読性） | AS-1（フィードバック反映でセキュリティに影響する場合のみ） |
| Phase 7（MVP） | 全視点インクリメンタル | 適用条件マトリクスに従い全視点を評価 | 適用条件マトリクスに従い全ドメインを評価 |
| Phase 8（フルスケール） | 全視点インクリメンタル | 適用条件マトリクスに従い全視点を評価 | 適用条件マトリクスに従い全ドメインを評価 |

### 品質チェック視点適用マトリクス（Phase 7-8）

| 視点 | 適用条件 | 非該当時の記載 |
|------|----------|--------------|
| RP-1: データ設計 | データモデル/スキーマの変更がスコープに含まれる場合 | 「本変更にデータモデル変更なし — N/A」 |
| RP-2: I/F 設計 | API/型定義の変更がスコープに含まれる場合 | 「本変更にI/F変更なし — N/A」 |
| RP-3: 冗長性排除 | **常時適用**（既存コードとの重複チェック） | — |
| RP-4: 変更耐性 | **常時適用** | — |
| RP-5: エラーハンドリング | **常時適用** | — |
| RP-6: パフォーマンス（UI体感） | UI 関連コードの変更がスコープに含まれる場合 | 「本変更にUI関連コード変更なし — N/A」 |
| RP-7: 可読性 | **常時適用** | — |

### 安全チェックドメイン適用マトリクス（Phase 7-8）

| ドメイン | 適用条件 | 非該当時の記載 |
|----------|----------|--------------|
| AS-1: セキュリティ | 認証・入力処理・データアクセス・外部連携の変更がある場合 | 「本変更にセキュリティ関連変更なし — N/A」 |
| AS-2: 安定性 | エラーハンドリング・リカバリ・データ整合性パスの変更がある場合 | 「本変更に安定性関連変更なし — N/A」 |
| AS-3: 可用性 | インフラ・スケーリング・リソース使用量の変更がある場合 | 「本変更に可用性関連変更なし — N/A」 |
| AS-4: 致命的パターン | **常時適用**（メモリリーク、無限ループ、デッドロック等） | — |

### トリビアル変更の判定

ランタイム動作に影響しない変更（コメント、フォーマット、ドキュメント、設定ファイルの非ランタイム変更）はインクリメンタルレビューをスキップできる。

```
[トリビアル変更の判定フロー]
├─ コーディングエージェントが「トリビアルスコープ」を宣言
│   └─ 宣言根拠: 変更がランタイム動作に影響しないことを説明
├─ コードレビュアーが分類を確認
│   ├─ 確認 → インクリメンタルレビューをスキップ（記録は残す）
│   └─ 却下 → 通常のインクリメンタルレビューを実施
```

## CONFLICT_RESOLUTION（判断の衝突時）

複数の視点で判断が矛盾する場合は、最上位原則「ユーザーの意図を完遂させる」に立ち返り、よりユーザー価値に貢献する方を選択する。判断の根拠を明記すること。

**コスト対効果の評価基準:**
- 改修コストと効果を比較する際、「効果」にはユーザー体感の改善価値を含めること
- 「コスト」を開発工数のみで評価し、ユーザー体感の劣化を放置することによる継続的な損失を無視してはならない
- 機能が動作するだけでは「ユーザーの意図の完遂」とは言えない。ユーザーがストレスなく使える状態をもって完遂とする
