# Phase Definitions（Phase 0-8）

全フェーズは完全なアジャイルで進行し、1日の中で何度も反復される。
課題や懸念があれば、どのフェーズからでも前のフェーズに立ち返って仕切り直す。

## Phase 0: 現状把握と目標設定

**目的:** 顧客業務の現状と理想の姿、直近で目指す目標地点を設定する

**整理すべき情報:**
1. 顧客の業界と事業内容
2. 顧客業務の現状（今どうやっているか）
3. 理想の姿（最終的にどうなりたいか）
4. 直近の目標地点（まず何を達成するか）

**壁打ち指針:**
- 業界の一般的な課題やトレンドについて壁打ちし視野を広げる
- 「現状」と「理想」のギャップを明確にする問いを投げる
- 直近の目標が現実的かどうかを検証する

**ゲート条件:**
- [ ] 顧客の業界・事業内容が明確
- [ ] 現状の業務フローが具体的に説明可能
- [ ] 理想の姿が描けている
- [ ] 直近の目標地点が設定されている
- [ ] 該当するドメインコンテキスト（`.ai-native/domain-context/`）があれば読み込み済み

**判定:** 上記4項目すべてにオペレーターが回答できていること。曖昧な場合は追加の問いで明確化を促す。ドメインコンテキストが存在する場合は、読み込みが完了していることを確認する。

**成果物出力先:** `outputs/phase0/current-state.md`

## Phase 1: 真の課題の追求

**目的:** 顧客の表面的な要望の裏にある真の課題を追求し、どのような成果をもたらしたいかを明確にする

**壁打ち指針:**
- 要望と実際の業務課題にギャップがないか検証する
- 「なぜそれが問題なのか？」を繰り返し掘り下げる（5 Whys）
- 「その課題が解決されたら、具体的に何が変わるのか？」を問う
- 課題の優先度を整理する

**ゲート条件:**
- [ ] 顧客の真の課題が1文で言語化できている
- [ ] その課題を解決することで得られる成果が明確

**判定:** 「このプロジェクトで解決する真の課題を1文で述べてください」の回答が以下を満たすこと:
- 具体的である（「業務を効率化する」のような抽象表現ではない）
- 顧客の業務文脈に根差している
- Phase 0の現状と理想のギャップを反映している

**成果物出力先:** `outputs/phase1/true-problem.md`

## Phase 2: 業務と実運用の明確化

**目的:** ユーザーの業務と実運用についてシステムで解決すべき課題を明確化する

**壁打ち指針:**
- ユーザーの1日の業務フローを時系列で整理する
- 各業務ステップで「何を入力し、何を出力するか」を明確にする
- 現状の業務で発生している非効率・ミス・ストレスを特定する
- システム化すべき箇所とシステム化すべきでない箇所を仕分ける
- 運用上の制約条件（使用環境、同時利用者数、利用頻度等）を確認する

**ゲート条件:**
- [ ] ユースケースが具体的なシナリオとして3つ以上
- [ ] 各ユースケースでユーザーの操作フロー（入力→処理→出力）が明確
- [ ] システムで解決すべき課題とシステム外で対応すべき課題が仕分け済み

**判定:** 「〇〇さんが△△の画面で□□を入力すると、××が表示される」レベルの具体性を求める。

**成果物出力先:** `outputs/phase2/usecases.md`, `outputs/phase2/business-flow.md`

## Phase 0-2 総合ゲート

Phase 3に進む前に以下の2条件を**同時に**満たすこと:

> **ゲート条件1:** 顧客の真の課題が1文で言語化できている
> **ゲート条件2:** ユースケースが具体的なシナリオとして3つ以上

## Phase 3: 要件定義

**目的:** Phase 0-2を踏まえた要件定義（業務要件・非機能要件）の作成

**業務要件:**
- 機能一覧（Phase 2のユースケースから導出）
- 各機能の入出力定義
- 業務ルール・制約条件
- ユーザー権限の定義

**非機能要件:**
- パフォーマンス基準（デフォルト: 初期表示200ms以下、検索100ms以下）
- セキュリティ要件（扱うデータの機密度に応じた対策レベル）
- 同時接続ユーザー数の想定
- 動作環境の制約
- 可用性要件

**壁打ち指針:**
- ユースケースから機能一覧を漏れなく導出できているか検証
- 非機能要件の数値基準を具体化
- 「この要件が漏れていると後で何が起きるか」の視点で抜け漏れ指摘

**ゲート条件:**
- [ ] 業務要件が機能一覧として整理されている
- [ ] 各機能の入出力が定義されている
- [ ] 非機能要件が数値または基準として定義されている
- [ ] 扱うデータの機密度が特定されている

**成果物出力先:** `outputs/phase3/functional-requirements.md`, `outputs/phase3/non-functional-requirements.md`

## Phase 4: 技術スタック確定

**目的:** 要件に最適な技術スタックを確定する

**検討項目:**
- フロントエンド（フレームワーク、UIライブラリ）
- バックエンド（言語、フレームワーク、API設計方針）
- データベース（RDB / ドキュメントDB / ファイル保管）
- インフラ（クラウド / オンプレミス、デプロイ構成）
- 認証・認可の方式
- CI/CD構成

**壁打ち指針:**
- 各技術選定の理由を「なぜそれを選んだのか」で掘り下げる
- 要件に対してオーバースペックまたはアンダースペックになっていないか検証
- 動作環境の制約との整合性を確認
- データの機密度に対して適切なインフラ構成か検証

**ゲート条件:**
- [ ] 全レイヤーの技術スタックが確定
- [ ] 各選定に対して要件ベースの理由が説明可能
- [ ] インフラ構成が非機能要件と整合

**成果物出力先:** `outputs/phase4/tech-stack-decision.md`

## Phase 5: 基本設計 + プロトタイプ開発

**目的:** 基本設計を進めながら同時にプロトタイプを開発する

**構造的特性:** Phase 0-4は線形進行だが、Phase 5は内部に反復ループを持つ。

```
I/F設計 → プロトタイプで検証 → 矛盾・不整合を検出 → I/F設計を修正 → 再検証
   ↑                                                                    │
   └────────────────────────────────────────────────────────────────────┘
```

終了条件: **全データフローがI/Fレベルで矛盾なく通ること**

**基本設計:**
- サイトマップの作成
- 画面ごとの機能定義
- I/F設計（API定義、モデル定義）

**I/F設計 6視点チェック:**
1. **技術スタック制約:** Phase 4で確定したDB種別・ファイル形式との整合性
2. **ユースケース:** Phase 2の壁打ちログからのユーザー利用シーン想定
3. **ユーザビリティ:** UIの最適な入出力表現に適合するデータ形式か（@ref:review-standards#USABILITY_STANDARDS §設計時ユーザビリティ U-1〜U-4 で詳細確認）
4. **データ設計上の都合:** マスタの静的/動的判断、select/autocomplete等のUI要素との連動
5. **型の継承関係:** class/interface/typeの継承・実装関係に矛盾がないか
6. **データフロー整合性:** 起点I/Fから派生する全I/Fが一気通貫で整合しているか

**データ設計原則:**
- データは正規化する。非正規化は例外措置（read/writeコストバランスの明確な根拠が必要）
- テーブルごとに意味を持たないユニークIDをPKとし、relationにはそのIDを使用
- 複合キーで強い制約を作らない（アジャイル開発中は変更に弱い）
- マスタデータは静的/動的を判断し、動的マスタにはCRUD機能を設計

**API設計原則:**
- 1つのAPIは1つの責務。癒着させない
- クライアント側に集約・加工の責務を押し付けない
- レスポンスに別リソースとして切るべきデータを混在させない
- API定義を見て使い方がわかる単位で切る

**プロトタイプ開発:**
- I/F設計と並行してダミーデータで全データフローを検証
- 矛盾・不整合はI/F設計にフィードバックし修正→再検証
- UI上で動作確認できるレベルまで作る

**インクリメンタルレビュー（軽量デザインレビュー）:** @ref:core-principles#SP-8 に基づき、実装タスク完了ごとにコードレビュアー（RP-1, RP-2 中心）→ システム監査官（AS-4 のみ）のインクリメンタルチェックを実施する。プロトタイプフェーズでは設計の整合性確認と致命的パターン検出に焦点を当てる。

**ゲート条件:**
- [ ] サイトマップが作成されている
- [ ] 画面ごとの機能定義がある
- [ ] I/F設計が6視点でチェック済み
- [ ] データ設計が正規化の原則に従っている
- [ ] API設計に癒着がない
- [ ] プロトタイプがダミーデータで動作
- [ ] 全データフローがI/Fレベルで検証済み

**差し戻し判断:** I/F設計やデータフローに重大な矛盾 → Phase 2（ユースケース）またはPhase 3（要件定義）に差し戻し。

**成果物出力先:** `outputs/phase5/architecture.md`, `outputs/phase5/screen-design.md`, `outputs/phase5/api-design.md`, `outputs/phase5/data-design.md`

## Phase 6: プロトタイプベースのフィードバック

**目的:** 業務に近い人（営業・顧客）とプロトタイプを中心にした対話を繰り返しフィードバックを反映する

**フィードバック収集のポイント:**
- ユーザーの業務フローに沿って操作してもらう
- 「何か違和感がありますか？」ではなく「普段の業務でこの画面をどう使いますか？」と聞く
- 操作中の迷い・戸惑いを観察する（言葉にならないフィードバックが重要）
- 機能の過不足だけでなく画面遷移やデータの表示順序も確認

**壁打ち指針:**
- フィードバックを整理し対応の優先度をつける
- Phase 0-2の前提を覆すフィードバック → 該当フェーズへの差し戻しを提案
- 「この変更を入れるとI/F設計に影響があるか」を検証

**ゲート条件:**
- [ ] キーパーソン（意思決定者）からプロトタイプへの承認が得られている
- [ ] 未反映のフィードバックがある場合、対応方針が合意されている
- [ ] フィードバック反映後のI/F設計に矛盾がない
- [ ] ユーザビリティ基準（@ref:review-standards#USABILITY_STANDARDS §検証時ユーザビリティ UX-1〜UX-5）で重大な問題がないこと

**成果物出力先:** `outputs/phase6/feedback-log.md`

## Phase 7: MVP構築

**目的:** 合意形成がなされたプロトタイプをベースに、基本機能を実装しMVPとする

**原則:** MVPに含める機能は「直近でユーザーが使用する機能」に限定する。

**インクリメンタルレビュー:** @ref:core-principles#SP-8 に基づき、実装タスク完了ごとにコードレビュアー（品質チェック）→ システム監査官（安全チェック）のインクリメンタルレビューを実施する。すべての指摘（ISSUE）が解消（CLEAR）するまでイテレーションを繰り返す。蓄積されたインクリメンタルレビュー記録はゲートレビューの入力となる。

**ゲートレビューの3層構造:** @ref:core-principles#SP-4 に従い、コードレビュアー（品質ゲート）→ システム監査官（安全ゲート）→ オペレーター判断の順で実施する。ゲートレビューは全コードベースを対象とし、インクリメンタルレビューの蓄積記録を参照しつつも独立した判定を行う。

**並行実装:** @ref:core-principles#SP-7 に基づき、独立した実装タスクはコーディングエージェントに並行で分配する。各実装の品質ゲート・安全ゲートは @ref:core-principles#SP-4 に従う。PMが並行タスクのコーディネーションを担う。

**進捗管理:** Phase 7以降の進捗管理はPM・スクラムマスターが進捗管理テンプレート（progress-management.md）で管理する。Phase 0-6の記録はオペレーターがナビゲーターのセッションサマリーを元に記録する（任意）。

**ゲート条件:**
- [ ] 直近でユーザーが使用する機能がすべて実装されている
- [ ] コードレビュアーの7視点レビュー（データ設計・I/F設計・コード品質）完了、オペレーターがサマリーを承認（品質ゲート通過）
- [ ] システム監査官の監査からリリースOK判断（安全ゲート通過）
- [ ] ユーザー・運用サポートのテストシナリオからリリースOK判断
- [ ] テクニカルライターによる主要機能のナレッジベースドキュメント（初版）が作成されている
- [ ] 本番運用を伴うシステムの場合、運用監視の可視化手段が確保されている（システム監査官 AS-3 で検証）
- [ ] 想定エラーにエラーコードが付与され、コードベースで一元管理されている

**判定:** 上記条件すべてがOKでなければMVPリリースを許可しない（適用条件付き項目は該当する場合のみ）。

**成果物出力先:** `outputs/phase7/dev-log.md`, `outputs/phase7/review-results.md`, `outputs/knowledge-base/`（ナレッジベース初版）

## Phase 8: フルスケール実装

**目的:** 6つのAIロールをフル稼働させて実装完了まで走る

**6ロールのフル稼働体制:**

| ロール | 稼働内容 |
|--------|---------|
| PM・スクラムマスター | 進捗管理、要件整合性確認、フィードバック共有 |
| コーディングエージェント | 実装の主体 |
| コードレビュアー | 実装タスク完了ごとのインクリメンタル品質チェック（SP-8）＋ ゲート判定時の全コードベース品質レビュー（品質ゲート） |
| システム監査官 | 実装タスク完了ごとのインクリメンタル安全チェック（SP-8）＋ ゲート判定時の全コードベース安全監査（安全ゲート） |
| ユーザー・運用サポート | テストシナリオの継続的実行、UX検証 |
| テクニカルライター | ナレッジベースの作成・改訂、サポートボットフィードバック反映 |

**並行実装:** Phase 7 と同様（@ref:core-principles#SP-7）。PMが並行タスクのコーディネーションと結果集約を担う。

**注意事項:**
- 速度が上がっても品質ゲート・安全ゲート・オペレーターのサマリー確認を省略しない
- システム監査官の指摘は最優先で対応（特にセキュリティとリソースコスト）
- ユーザー・運用サポートからのフィードバックがPhase 0-2の前提を覆す場合は差し戻す勇気を持つ

**ゲート条件:**
- [ ] 要件定義された全機能が実装されている
- [ ] コードレビュアーの最終レビュー完了、オペレーターがサマリーを承認（品質ゲート通過）
- [ ] システム監査官の最終監査パス（安全ゲート通過）
- [ ] ユーザー・運用サポートの全テストシナリオパス
- [ ] リリース前のエラーハンドリング設計が開発・運用サポート視点で議論・解決済み
- [ ] テクニカルライターによるナレッジベースの網羅性検証済み（全機能カバー）
- [ ] ユーザー・運用サポートによるナレッジベース品質検証パス（「このドキュメントでユーザーが自己解決できるか」）
- [ ] サポートボットを用いたリリース前テストで人間がサポートボットの回答品質を確認済み
- [ ] 本番運用を伴うシステムの場合、運用監視の可視化手段が確保され、ユーザーアクションログの設計がAS-1要件と整合している（システム監査官 AS-3 で検証）
- [ ] エラーコード逆引きリファレンス（`ops-guide/error-code-reference.md`）がテクニカルライターにより作成されている

**成果物出力先:** `outputs/phase8/dev-log.md`, `outputs/phase8/review-results.md`, `outputs/phase8/release-note.md`, `outputs/knowledge-base/`（ナレッジベース完成版）

## EMERGENCY_PATH（緊急対応パス）

本番環境で重大な障害やセキュリティ脆弱性が発見された場合、9フェーズフローを圧縮した緊急対応パスを使用する。

### 適用条件

以下のいずれかに該当する場合に限り、緊急対応パスを使用できる:
- **セキュリティ脆弱性:** 本番環境で悪用可能な脆弱性が発見された
- **サービス停止:** ユーザーがシステムを利用できない状態
- **データ損失リスク:** データの破損・消失が進行中または差し迫っている

### 緊急対応フロー

```
1. トリアージ（オペレーター）
   → 重大度判定: Critical（即時対応）/ High（24時間以内）
   → 影響範囲の特定
   → 緊急対応パスの適用を宣言（意思決定ログに記録）

2. 最小スコープ修正（コーディングエージェント）
   → 問題の根本原因を特定
   → 最小限の変更で修正（スコープ拡大しない）
   → 修正が影響する範囲を明示

3. 二重ゲートレビュー（圧縮版）
   → システム監査官: 安全ゲート（省略不可。修正が新たなリスクを生まないか）
   → コードレビュアー: 品質ゲート（修正の技術的妥当性。7視点のうち関連視点のみ）
   → 両ゲートを並行実施可（通常フローの直列実施の例外）

4. オペレーター承認 → デプロイ

5. ポストモーテム（48時間以内）
   → なぜ発生したか（根本原因分析）
   → なぜ通常フローで防げなかったか
   → 恒久対応が必要か → 必要なら通常フェーズフローに Issue を起票
```

### 緊急対応パスのゲート条件

| # | 条件 | 省略可否 |
|---|------|---------|
| E-1 | 修正が問題を解決する（検証済み） | 省略不可 |
| E-2 | 修正が新たなセキュリティリスクを生まない（安全ゲート） | 省略不可 |
| E-3 | 修正の技術的妥当性が確認されている（品質ゲート） | 省略不可 |
| E-4 | 修正の影響範囲が明示されている | 省略不可 |
| E-5 | ポストモーテムが計画されている | 省略不可 |

### 記録

緊急対応は通常フロー以上に追跡可能性が重要。以下を必ず記録する:
- 意思決定ログ: 緊急対応パス適用の判断とその理由
- フェーズ遷移履歴: 「緊急対応パス適用」として記録
- ポストモーテム結果: `outputs/emergency/postmortem-YYYY-MM-DD.md`

## SCOPE_CLASSIFICATION（スコープ分類）

すべての作業が9フェーズフルフローを必要とするわけではない。作業スコープに応じて適切なフローを選択する。

### スコープ階層

| スコープ | 定義 | フロー | 必要ロール |
|---------|------|--------|-----------|
| **Full** | 新規プロジェクト、または既存システムの大規模機能追加 | Phase 0-8 全フェーズ | 全ロール（フェーズに応じて段階的に） |
| **Medium** | 既存システムへの中規模機能追加（既存の設計・要件に増分する変更） | Phase 3から開始、Phase 5圧縮、Phase 7-8統合 | coding-agent + code-reviewer + system-auditor + オペレーター |
| **Minimal** | バグ修正、小規模変更、設定変更 | 実装 → 二重ゲートレビュー → オペレーター承認 | coding-agent + code-reviewer + system-auditor + オペレーター |

### Medium スコープのフロー

```
前提: Phase 0-2 の既存成果物（ユースケース、真の課題）が存在すること

Phase 3（圧縮版）: 既存要件への増分として要件を定義
  → 既存の非機能要件・データ機密度を継承
Phase 4（省略可）: 技術スタックが変わらない場合はスキップ
Phase 5（圧縮版）: 既存設計への増分としてI/F設計をレビュー
  → プロトタイプは変更箇所の動作確認のみ
Phase 7-8（統合）: 実装 → SP-8 インクリメンタルレビュー → 二重ゲートレビュー → オペレーター承認
  → インクリメンタルレビュー: 実装タスク完了ごとに CLEAR/ISSUE で品質・安全チェック
  → ゲートレビュー: 蓄積されたインクリメンタル記録を参照しつつ全体を判定
```

### Minimal スコープのフロー

```
1. coding-agent: 修正を実装
2. code-reviewer: 品質ゲート（変更差分に関連する視点に絞って実施）
3. system-auditor: 安全ゲート（変更がセキュリティ・安定性に影響する場合のみフル監査）
4. オペレーター: 承認
```

> **SP-8 インクリメンタルレビューとの関係:** Minimal スコープではインクリメンタルレビューは不要。ゲートレビュー自体が変更差分に対して直接実施されるため、インクリメンタルレビューと実質的に同等のカバレッジとなる。

**Minimal スコープの適用条件:**
- 既存の設計・アーキテクチャを変更しない
- 新しいデータエンティティを追加しない
- 新しい外部システム連携を追加しない
- 上記いずれかに該当する場合は Medium 以上を適用する

### スコープ判定の記録

スコープの判定はオペレーターが行い、意思決定ログに記録する。判定を誤った場合（Minimal で始めたが設計変更が必要になった等）は、適切なスコープにエスカレーションする。

## PHASE_JUMP_RULES（フェーズ間差し戻しルール）

どのフェーズからでも以下に該当する場合は前のフェーズに差し戻す:

| 検知した問題 | 差し戻し先 |
|-------------|-----------|
| 顧客の真の課題の認識が変わった | Phase 1 |
| 新しいユースケースが発見された | Phase 2 |
| 要件の追加・変更が必要 | Phase 3 |
| 技術スタックの変更が必要 | Phase 4 |
| I/F設計に重大な矛盾 | Phase 5 |
| 関係者の合意が崩れた | Phase 6 |

差し戻し時は「なぜ戻るのか」「何を解決して戻ってくるのか」を明確にする。

## TWO_LAYER_GATE_SYSTEM（2層ゲートシステム）

すべてのPhaseのゲート判定は以下の2層で実施する。

### 第1層: 静的チェック

phase-definitions.mdに明記されたゲート条件を1つずつ検証する。

- 各条件に対してPASS / NOT YETの二択で判定する
- 「だいたいOK」は許可しない
- 満たしていない場合は何が不足しているかを具体的に示す

### 第2層: 生成的チェック

AIが以下の3つの問いに基づき、静的チェックリストでは検出できない抜け漏れを推論する。

**生成的チェックの3つの問い:**

1. **次Phase前提条件チェック:** Phase {N+1} が正常に開始できるために、Phase {N} の成果物に不足している情報はないか？
2. **methodology整合性チェック:** methodology全体の定義・原則と照合して、定義されているが成果物に反映されていない項目はないか？
3. **実運用観点チェック:** 実運用の観点から、明記されていないが本来必要な決定事項はないか？

**過剰検出の抑制:**
- 指摘は「これがないと具体的に何が困るか」を説明できるもののみ挙げること
- 推測や「あると良い」レベルの指摘は含めない

### 判定フロー

```
1. 第1層（静的チェック）を実施 → 全条件 PASS / NOT YET を判定
2. 第1層が全PASS → 第2層（生成的チェック）を実施
3. 第2層で検出された項目をオペレーターに提示
4. オペレーター判断:
   a. 指摘が妥当 → 対応してからゲート再判定
   b. 指摘が過剰 → オペレーターがフィルタし、理由を記録してゲート通過
   c. 検出項目なし → ゲート通過
```

### 2層ゲートの目的

従来の静的チェックリストのみでは「チェックリスト自体の不備」を検出できない。
エンジニアが感じる「後になって要件が増えた」「今更要件変更か」という問題の多くは、
上流Phaseのゲートで検出すべき抜け漏れが通過してしまったことに起因する。
第2層の生成的チェックにより、この構造的な問題を軽減する。

## WALLBOUNCING_STANCE（壁打ちの姿勢）

### Phase 0-2: 引き出す姿勢（オペレーター優位）

オペレーターは業界特性、既存システム、実業務の癖といったドメイン知識を持っている。

- 答えを与えるのではなく、問いで思考を引き出す
- 「なぜそう判断したのか？」を常に問い、判断の根拠を言語化させる
- 「顧客の真の課題」に到達するまで妥協しない
- オペレーターが明らかに誤った方向に進んでいる場合は明確に指摘する
- 選択肢で回答可能な場面は選択肢形式を使い、ドメイン知識の引き出しが必要な場面は例示付きの自由記述を使う（@ref:core-principles#SP-6）

### Phase 3-5: 提案する姿勢（AI優位）

- 技術的な選択肢を比較・提示し、それぞれのトレードオフを明確にする
- 「どの技術をどこに使うか」「データをどう持つか」の最終判断はオペレーターに委ねる
- オペレーターの判断に技術的リスクがある場合はリスクを具体的に示し再考を促す
- 技術的選択肢は番号付きで提示し、各選択肢のトレードオフを明記する。「その他（自由記述）」を常に含める（@ref:core-principles#SP-6）

### Phase 6: 整理・検証する姿勢（対等）

フィードバックを整理・構造化し、設計への影響を検証する姿勢。

- 収集したフィードバックを整理し、対応の優先度をつける
- Phase 0-2の前提を覆すフィードバックを検知したら、差し戻しを提案する
- 「この変更を入れるとI/F設計に影響があるか」を検証する
- 業務フローに沿って操作してもらい、言葉にならないフィードバック（操作中の迷い・戸惑い）を重視する
- 選択肢で優先度付けを促し、影響分析結果は具体的に提示する（@ref:core-principles#SP-6）

### ゲート判定の姿勢

- 完了条件を1つずつチェックし、すべて満たされているか確認する
- 「だいたいOK」は許可しない。明確に満たしているか/いないかの二択
- 満たしていない場合は何が不足しているかを具体的に示し、解決のための問いを投げる
- 忖度しない。条件を満たしていないのに「良いですね」と言うことは絶対にしない
- 第1層（静的チェック）通過後、必ず第2層（生成的チェック）を実施する（TWO_LAYER_GATE_SYSTEM参照）
- 第2層で検出された項目はオペレーターに提示し、オペレーターが最終判断する
- NOT YET 判定時の問いかけは、解決アプローチを選択肢形式で提示する（@ref:core-principles#SP-6）
