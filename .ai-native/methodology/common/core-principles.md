# Core Principles

## TOP_LEVEL_PRINCIPLE

**「ユーザーの意図を完遂させる」**

- すべてのフェーズ、すべての判断、すべてのレビューはこの原則に帰結する
- 判断が衝突した場合は「どちらがよりユーザーの意図の完遂に貢献するか」で優先度を決定する
- この原則は例外なく最優先であり、いかなる技術的判断・プロセス判断よりも上位に位置する

---

## STRUCTURAL_PRINCIPLES（構造原則）

### SP-1: オペレーターは意思決定者である

- オペレーター = 作業者ではなく「判断・承認・差し戻し」の責任者
- 従来の開発チームにおけるテックリード・スクラムマスター・PM・企画者の判断責務を1人に集約した存在
- AIの出力を無条件で受け入れることは決してしない
- エンジニア経験3年以上を前提とする（AIの出力の妥当性を評価できる能力が必要）

**オペレーターがやること:**
- 顧客の真の課題の見極め
- 技術選定の最終判断
- フィードバックの優先度付け
- AIレビュー結果サマリーの妥当性判断

**オペレーターがやらないこと:**
- 実装の主体にはならない（→ コーディングエージェント）
- 監査の主体にはならない（→ システム監査官）

**DELEGATION_PROTOCOL（委任プロトコル）:**

オペレーターが不在となる場合、以下の手順で権限を委任できる:

1. **代理の指名:** オペレーターは同等の権限を持つ代理を指名し、委任期間を明示する。代理の要件はオペレーターと同等（エンジニア経験3年以上、ドメイン知識を有する）
2. **記録:** 委任の開始・終了を `progress-management.md` のオペレーター/代理追跡欄に記録する
3. **複数オペレーター体制:** チーム開発では、リードオペレーターが最終判断権を持つ。オペレーター間の判断不一致は意思決定ログに記録し、リードオペレーターが決定する

### SP-2: AIロールは統合しない

ロール分離は効率的な分業のためではなく、**相互牽制による品質保証のための構造設計**である。
プロジェクト内で稼働する6つのロールに加え、方法論そのものを進化させるメタロール（方法論エデュケーター）が存在する。

| ロール | 責務 | 主要稼働フェーズ |
|--------|------|-----------------|
| PM・スクラムマスター | 進捗管理、要件整合性確認、フィードバック共有 | Phase 7-8 |
| コーディングエージェント | 実装の主体 | Phase 5以降 |
| コードレビュアー | 7つのレビュー視点に基づくデータ・設計・コード品質レビュー（品質ゲート） | Phase 5-8（Phase 5-6はインクリメンタルモード、Phase 7-8はインクリメンタル+ゲート） |
| システム監査官 | 安全性・安定性・可用性に特化した独立監査（安全ゲート） | Phase 5-8（Phase 5-6はインクリメンタルモード、Phase 7-8はインクリメンタル+ゲート） |
| ユーザー・運用サポート | ペルソナベースのテストシナリオ策定・UX検証 | Phase 6以降 |
| テクニカルライター | サポートボット用RAGナレッジベースの作成・維持 | Phase 7-8 |
| 方法論エデュケーター | 方法論そのものの評価・改善・教育（メタロール） | 全Phase（プロジェクト横断） |

**分離を維持する3つの理由:**
1. **異質性の担保:** 同一AIが実装と監査を兼ねると、自身の出力に対する批判的検証が構造的に弱化する
2. **相互牽制の実効性:** 牽制は独立性があって初めて機能する。利益相反は牽制を形骸化させる
3. **盲点の非重複:** 異なるロールは異なる盲点を持つ。分離により単一視点では検出できない問題を構造的に捕捉する

**分離が必要な具体的理由:**
- PM vs コードレビュアー: PMは「進捗を前に進める」、レビュアーは「品質基準に達するまで止める」→ 本質的に相反する責務
- コードレビュアー vs システム監査官: 「設計が正しいか・コードがきれいか」vs「安全か・壊れないか・止まらないか」→ 品質レンズと安全レンズの分離。セキュリティ・リソースコストは安全ゲートに一本化し、責務の重複を排除する

### SP-3: ゲート判定は2層で実施する（2層ゲートシステム）

> **緊急対応時の例外:** 本番障害時は `phase-definitions.md` の EMERGENCY_PATH に定義された圧縮フローを使用する。安全ゲートは緊急時も省略不可。

すべてのPhaseのゲート判定は以下の2層で実施する:

| 層 | 方式 | 目的 |
|---|------|------|
| **第1層: 静的チェック** | phase-definitions.mdに明記された条件を1つずつ検証 | 既知の条件を漏れなく確認 |
| **第2層: 生成的チェック** | AIが「次Phaseの前提条件」「methodology整合性」「実運用観点」から抜け漏れを推論 | チェックリスト自体の不備を検出 |

- **判定者はAI。** 人間（オペレーター）は差し戻し権限を持つ
- 第1層が全PASS後に第2層を実施する
- 第2層の検出項目はオペレーターに提示し、妥当/過剰をオペレーターが判断する
- 過剰検出の抑制: 「これがないと具体的に何が困るか」を説明できる項目のみ挙げる

**なぜ2層が必要か:**
エンジニアが感じる「後になって要件が増えた」「今更要件変更か」という問題の多くは、
上流Phaseのゲートで検出すべき抜け漏れが通過してしまったことに起因する。
静的チェックリストのみでは「チェックリスト自体の不備」を検出できない。
第2層の生成的チェックにより、この構造的な問題を軽減する。

詳細は `phase-definitions.md` の TWO_LAYER_GATE_SYSTEM セクションを参照。

### SP-4: レビューは2つの独立したゲートを通過させる

```
コーディングエージェント → 実装を出力
    ↓
コードレビュアー → 品質ゲート（7つのレビュー視点: データ設計・I/F設計・コード品質）
    ↓
システム監査官 → 安全ゲート（安全性・安定性・可用性）
    ↓
オペレーター → 両方のサマリーを読み、承認 or 差し戻しを判断
```

- どちらか一方のゲートの省略は許可しない
- オペレーターはコードを逐行レビューするのではなく、AIのレビュー結果サマリーの妥当性を判断する

> **直列実行の根拠:** 品質ゲートFAILの場合、コードが修正される可能性が高く、安全ゲートの実施は非効率となる。安定したコードベースに対して監査を行うことで、安全ゲートの精度と効率を確保する。緊急時は修正スコープが限定的であるため、この前提が緩和され並行実施が許可される（EMERGENCY_PATH参照）。

### SP-5: 壁打ちの姿勢はフェーズで変える

| フェーズ | 優位性 | 姿勢 | 理由 |
|---------|--------|------|------|
| Phase 0-2 | オペレーター優位 | 引き出す | オペレーターが持つドメイン知識（業界特性、既存システム、実業務の癖）をAIが引き出す |
| Phase 3-5 | AI優位 | 提案する | AIの技術的知見を活かし選択肢を提示。ただし最終判断はオペレーター（ドメイン知識が判断を左右する） |
| Phase 6 | 対等 | 整理・検証する | フィードバックの優先度付け・I/F設計への影響検証。「引き出す」でも「提案する」でもなく、収集された情報を構造化する |

### SP-6: 選択肢ベースのインタラクション（Guided Prompt）

オペレーターへの問いかけは、自由記述が必須でない限り、**番号付き選択肢 + 「その他（自由記述）」** の形式で提示する。

**原則:**
- 判断・選択を求める場面では、具体的な選択肢を番号付きで提示する
- 選択肢が3つ以上の場面では「その他（自由記述）」を最終選択肢として必ず含める
- YES/NO の二択判断では「その他」の代わりに補足記述欄を設け、判断の背景や条件付き回答を受け入れる
- オペレーターは番号で回答できる。複数選択が可能な場合はその旨を明示する

**適用基準:**

| 場面 | 方式 | 例 |
|------|------|---|
| 選択肢が列挙可能な判断 | 番号付き選択肢 | 技術スタック選定、優先度判断、対応方針の決定 |
| ドメイン知識の引き出し | 自由記述 + 例示で誘導 | 業務フローの説明、課題の言語化 |
| YES/NO 判断 | 二択 + 補足記述欄 | ゲート条件の確認、差し戻し判断 |

**自由記述が必要な場面でも「例示」で思考を誘導する:**
- 「例えば〇〇のような課題はありますか？」と具体例を添える
- 「他のケースでは△△が課題になることが多いですが、いかがですか？」と類似事例で誘導する

**フォーマット例:**

```
この機能の優先度をどう判断しますか？

1. 必須（これがないと業務が止まる）
2. 重要（あると業務効率が大幅に上がる）
3. あると良い（なくても業務は回る）
4. 不要（今回のスコープ外）
5. その他（自由記述）
```

### SP-7: アドホックロールオーケストレーション

オペレーターが特定のテーマについて議論・検討・タスク実行を求めた場合、ナビゲーターは必要なロールをアドホックに招集し、マルチロール協議または並行タスク実行を組織できる。

> **SP-8 との関係:** アドホック招集による並行タスクの各完了物も、SP-8 のインクリメンタルレビューパイプラインの対象となる。

**フェーズフローとの関係:**
- アドホック招集はフェーズ進行とは独立した意思決定支援プロセスである
- フェーズベースのロール活性化（SP-2のロール表）およびレビューの直列パイプライン（SP-4）は変更しない
- アドホック招集の結果はフェーズの成果物に反映されるが、ゲート条件の代替にはならない

**アドホック招集の原則:**
1. オペレーターのリクエストに基づきナビゲーターが招集判断を行う
2. 招集されたロールはSP-2の役割分離を維持する（招集されても統合しない）
3. 招集結果はオペレーターに報告され、最終判断はオペレーターが行う（SP-1）

**並行タスク実行の原則:**
1. タスク間にデータ依存性・成果物の前提依存がない場合、複数ロールで並行実行できる
2. ナビゲーターがコーディネーター/アグリゲーターとして依存関係を分析し、結果を集約する
3. 同一ロールの複数インスタンス化は効率性が求める場合に許容する
4. コーディングエージェントの独立した実装タスクは、フェーズフロー内（Phase 7-8）でも非同期並行で分配する
5. ただし各実装の品質ゲート・安全ゲートはSP-4に従い直列で実施する

**コスト評価の原則:**
- アドホック協議でコスト対効果を評価する際、「コスト」には開発工数だけでなく**ユーザー体感の劣化コスト**を明示的に含めること
- ユーザー体感の劣化（パフォーマンス低下、操作ストレス、停止感）は技術的負債と同等かそれ以上の隠れたコストである
- 「改修コスト＞効果」と結論づける場合、効果側にユーザー体感の改善価値が含まれているかを検証すること
- クリティカルなパフォーマンス問題は IQ-1 の「軽微な低下」の許容範囲外であり、改修コストの大小にかかわらず対処を検討すべき対象である

**思考過程の透明性:**
- ロール間の対話・推論過程はオペレーターに可視化する
- **サマリー層**（結論・推奨事項・オペレーター判断事項）と**詳細層**（議論経過・各ロールの見解・対立点）の2層構成で出力する
- サマリー層は判断に必要な情報を簡潔に提示し、詳細層は必要時に参照できる形で提供する
- 詳細層は方法論エデュケーターの改善データとしても機能する

### SP-8: インクリメンタルレビューパイプライン

実装タスク完了ごとに、変更範囲に対してコードレビュアー（品質チェック）→ システム監査官（安全チェック）を実施し、すべての指摘が解消するまでコーディングとレビュー/監査を繰り返す。ゲート判定時には全コードベースに対する正式なレビュー/監査を別途実施する。

> **既存原則との関係:** SP-4（2ゲート直列）の品質ゲート→安全ゲートの順序はインクリメンタルモードでも維持する。SP-2（ロール分離）により、コーディングエージェントの自己レビューではなく独立したレビュアー/監査官が実施する。

**なぜインクリメンタルレビューが必要か:**
- セルフチェック（CLAUDE.md Push 前チェック）は自己レビューであり、SP-2 の役割分離の精神を実装レベルで満たさない
- ゲートレビューのみでは品質問題の検出がフェーズ末期に集中し、修正コストが増大する
- 実装直後のレビューは変更の文脈が新鮮なうちに問題を検出でき、修正が最小コストで済む

**インクリメンタルレビューとゲートレビューの区別:**

| 観点 | インクリメンタルレビュー | ゲートレビュー |
|------|------------------------|--------------|
| スコープ | 変更範囲のみ | 全コードベース |
| 深度 | 適用条件に基づく選択的 | 全視点・全ドメイン |
| 承認 | なし（蓄積のみ。オペレーター関与不要） | オペレーター承認必須 |
| 判定値 | CLEAR / ISSUE | PASS / FAIL |
| 目的 | 品質の早期確保・左シフト | リリース判定 |
| 適用フェーズ | Phase 5〜8 | Phase 7-8 ゲート条件 |

**トリガー:** 実装タスク完了報告（`coding-agent.md` OUTPUT_FORMAT）の提出時。コミット単位ではなく、コーディングエージェントが完了と報告したタスク単位。

**イテレーションループ:**

```
コーディングエージェント: 実装タスク完了報告
    │
    ▼
コードレビュアー: インクリメンタル品質チェック（変更スコープ限定）
    │
    ├── CLEAR → システム監査官: インクリメンタル安全チェック（変更スコープ限定）
    │              │
    │              ├── CLEAR → タスク「インクリメンタル通過」（ゲートレビュー用に蓄積）
    │              │
    │              └── ISSUE → コーディングエージェントが修正 → 再提出
    │                          （品質に影響する修正の場合は品質チェックも再実施）
    │
    └── ISSUE → コーディングエージェントが修正 → 再提出（ループ再開）
```

**蓄積されたインクリメンタルレビュー記録の活用:**
- ゲートレビュアーの入力として参照（既に検査済みの領域の把握）
- 繰り返し指摘されたパターンの可視化（コーディング規約候補。方法論エデュケーターの D-5 分析対象）
- 品質の経時的変化の追跡（オペレーターへの可視性）

**トリビアル変更の除外:** コメント・フォーマット・ドキュメントのみの変更など、ランタイム動作に影響しない変更はインクリメンタルレビューをスキップできる。コーディングエージェントが「トリビアルスコープ」を宣言し、レビュアーがその分類を確認または却下する。

詳細なスコープ定義・視点適用ルールは `review-standards.md` INCREMENTAL_REVIEW_SCOPE セクションを参照。
出力テンプレートは `review-output-template.md` §4 を参照。

## DOCUMENTATION_VISUALIZATION_RULES（ドキュメント図解ルール）

全ロールが生成・改訂するドキュメントにおいて、人間への可視性を高めるために **Mermaid 記法の図解** を必要に応じて活用する。

### 適用基準

以下のいずれかに該当する場合、テキストのみの説明ではなく Mermaid 図解を併用する:

| 対象 | 図解の種類 | 例 |
|------|-----------|-----|
| プロセス・手順の流れ | `flowchart` / `sequenceDiagram` | ゲート判定フロー、レビュープロセス、デプロイ手順 |
| ロール間・コンポーネント間の関係 | `flowchart` / `graph` | ロール間インタラクション、ハンドオフの流れ、API連携 |
| 時系列の変遷・履歴 | `timeline` / `gitgraph` | フェーズ遷移履歴、方法論バージョンの変遷 |
| 状態遷移 | `stateDiagram-v2` | フェーズ遷移、タスクステータス、ドキュメント状態 |
| 構造・階層の俯瞰 | `graph TD` / `mindmap` | アーキテクチャ構成、ドキュメント間の依存関係 |
| 比較・評価の分布 | `quadrantChart` / `xychart-beta` | 優先度マッピング、リスク評価 |

### 図解作成の原則

1. **補完であって代替ではない:** 図解はテキスト説明を補完するものであり、図だけで完結させない。図の前後に要約や補足テキストを配置する
2. **シンプルさの維持:** 1つの図に詰め込みすぎない。ノード数が多い場合は図を分割する
3. **日本語ラベルの使用:** 図中のラベル・説明は日本語で記述する（対象読者に合わせる）
4. **標準記法の遵守:** Mermaid 公式ドキュメントの記法に従い、非標準の拡張や廃止予定の構文を使用しない

### 適用しない場合

- 単純なリスト・表で十分に伝わる情報（図解による冗長化を避ける）
- 1対1の単純な対応関係（テキストや表が適切）
- 図解を入れることで文書の主旨がぼやける場合

## IMPLEMENTATION_QUALITY_PRINCIPLES（実装品質原則）

> **SoT 注記:** 本セクションの実装品質原則は方法論全体に適用される。Claude Code 環境では `CLAUDE.md` の「開発原則」セクションが Claude Code 固有の実装ルールとして追加適用される。両者に矛盾がある場合は本ドキュメント（methodology）の定義を優先する。

### IQ-1: 汎用化はユーザー価値に貢献する場合のみ
- 開発効率性・システム拡張性・可用性に貢献がある場合 → パフォーマンスの軽微な低下を許容して積極的に採用
- 用途のない抽象化は可読性を落とす → 入れない

### IQ-2: 他者が読んで意図がわかるか
- コード、設計、ドキュメントすべてにおいて「配慮」を求める
- マジックナンバー、意味のない変数名、意図不明なフォルダ構成は「配慮の欠如」

### IQ-3: レビュー前の前提条件
- エラーハンドリングの設計はレビュー提出前に完了していること
- 分岐の将来性（種類が増えるかどうか）はレビュー前に確認済みであること
- 「とりあえず動く」状態でレビューに出さない

## CHECK_AND_BALANCE（牽制関係マップ）

| 牽制の方向 | 内容 |
|-----------|------|
| コーディングエージェント → コードレビュアー | 実装の出力を7つのレビュー視点でデータ・設計・コード品質検証（品質ゲート） |
| コーディングエージェント → システム監査官 | 実装の出力を安全性・安定性・可用性で独立検証（安全ゲート） |
| PM・スクラムマスター → コーディングエージェント | 要件との整合性、進捗の妥当性を確認し逸脱を検知 |
| ユーザー・運用サポート → PM・スクラムマスター | PMが定義した要件・優先度をエンドユーザー視点で検証 |
| システム監査官 → 全ロール | セキュリティ・リソースコストの指摘は全ロールに対して最優先で適用 |
| PM・スクラムマスター → 全ロール | 進捗管理テンプレートで意思決定・変遷を可視化し、全ロールおよび第三者が追跡可能にする |
| テクニカルライター → ユーザー・運用サポート | 作成したナレッジベースの品質検証を依頼し、「ユーザーが自己解決できるか」の判断を得る |
| ユーザー・運用サポート → テクニカルライター | テストシナリオ・UX検証結果をナレッジベースの入力として提供し、品質検証でフィードバックする |
| 方法論エデュケーター → 全ロール | 各ロールの実践品質を観察し、方法論の有効性を評価。改善提案をオペレーター承認のもとで方法論に反映する |
| 壁打ちナビゲーター → 全ロール | アドホック協議（SP-7）において必要なロールを招集し、マルチロール協議を主催する。並行タスクのコーディネーションを行う（Phase 0-6） |
| PM・スクラムマスター → 全ロール（アドホック） | Phase 7-8においてナビゲーターからアドホック招集・並行タスクコーディネーションの責務を引き継ぐ（SP-7） |
| コードレビュアー → コーディングエージェント（インクリメンタル） | 実装タスク完了ごとに変更範囲の品質チェックを実施し、ISSUE があればコーディングエージェントに差し戻す（SP-8） |
| システム監査官 → コーディングエージェント（インクリメンタル） | 品質チェック CLEAR 後に変更範囲の安全チェックを実施し、ISSUE があればコーディングエージェントに差し戻す（SP-8） |

## OPERATOR_REVIEW_CHECKPOINTS（オペレーターのレビュー判断ポイント）

1. **7視点すべてがサマリーに含まれているか** — 視点の欠落がないか確認（「データ設計に言及なし」＝未検証の可能性）
2. **各視点の判定根拠が具体的か** — 「問題なし」だけでは不十分。何を確認して問題なしと判断したかが明示されているか
3. **指摘事項の重大度判定が妥当か** — セキュリティ・リソースコストの問題が軽微扱いされていないか
4. **ドメイン知識から見て見落としがないか** — 業界固有の制約、レガシーシステムとの互換性等
5. **根拠のスポットチェック（ハルシネーション検知）** — AIのレビュー結果が引用するファイルパス・行番号・コード構造のうち、最低1件を実物と照合する。引用が存在しないか、内容が異なる場合はレビュー全体を再実施させる
