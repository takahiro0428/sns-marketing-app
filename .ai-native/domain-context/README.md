# Domain Context（ドメインコンテキスト）

プロジェクト横断で使い回せるドメイン知識を管理するディレクトリ。
AIロール（特に壁打ちナビゲーター）の起動時に、該当するドキュメントを手動で読み込ませることで、業務理解に基づいた質の高い壁打ちを実現する。

---

## なぜドメインコンテキストが必要か

業務フローを理解しないまま画面設計やAPI開発を進めると、ユーザーに公開した時点で「使い物にならない」と突き返され、大規模な手戻りが発生する。Phase 0-2の壁打ちだけでドメイン知識を引き出すには限界がある。事前に文書化されたドメイン知識をAIに読み込ませることで、壁打ちの精度が上がり、手戻りリスクを低減できる。

---

## ディレクトリ構成

```
domain-context/
├── README.md          # 本ファイル（利用ガイド）
├── industry/          # 業界知識
├── business-flow/     # 業務フロー
└── persona/           # ペルソナ（利用者像・ペインポイント）
```

---

## カテゴリ定義

### industry/（業界知識）

対象業界の商慣習、規制、専門用語、業界特有の課題やトレンドを記述する。

**配置するドキュメント例:**
- 業界の商慣習・取引フロー
- 適用される法規制・コンプライアンス要件
- 業界固有の専門用語集
- 業界の一般的な課題・トレンド

**ファイル命名規則:** `{業界名}-{内容}.md`（例: `logistics-regulations.md`, `healthcare-terminology.md`）

### business-flow/（業務フロー）

ユーザーの実際の業務フローを記述する。日常業務の流れ、各業務ステップの入出力、発生している非効率やミスなど。

**配置するドキュメント例:**
- ユーザーの1日の業務フロー（時系列）
- 各業務ステップの入力・処理・出力
- 現状の業務で発生している非効率・ミス・ストレス
- 部署間・システム間のデータの流れ

**ファイル命名規則:** `{業務名}-flow.md`（例: `order-management-flow.md`, `inventory-check-flow.md`）

### persona/（ペルソナ）

システムの利用者像、利用シーン、ペインポイントを記述する。

**配置するドキュメント例:**
- 利用者の役職・スキルレベル・ITリテラシー
- 利用シーン（いつ、どこで、何のために使うか）
- 現状のペインポイント（何に困っているか）
- システムに対する期待値

**ファイル命名規則:** `{ペルソナ名}-persona.md`（例: `warehouse-manager-persona.md`, `field-worker-persona.md`）

## 使い方

### 1. ドキュメントの作成・配置

該当するカテゴリのディレクトリにMarkdownファイルを配置する。一度作成すれば複数プロジェクトで使い回せる。

### 2. AIロール起動時に読み込ませる

Phase 0開始前または壁打ち開始時に、該当するドメインコンテキストをAIセッションに読み込ませる。

**読み込ませるタイミング:**
- Phase 0（現状把握と目標設定）の開始前 — 必須
- 壁打ちナビゲーターの新規セッション開始時 — セッション再開プロンプトに含める

**読み込ませる方法:**
- Claude Projects: Project Knowledgeにアップロード
- Claude Code: セッション開始時にファイルパスを指定して読み込ませる
- ChatGPT / Gemini: セッション開始時にファイル内容を貼り付ける

### 3. Phase 0ゲート条件での確認

Phase 0のゲート条件に「該当するドメインコンテキストがあれば読み込み済みであること」が含まれている。ドメインコンテキストが存在する場合は、読み込みを確認してからPhase 1に進む。

---

## 注意事項

- ドメインコンテキストは**参考資料**であり、壁打ちの代替ではない。読み込んだ上でオペレーターとの壁打ちを通じて理解を深める
- ドメインコンテキストの内容が古くなった場合は、プロジェクト完了後に更新する
- 機密情報を含む場合は、リポジトリの公開範囲に注意する
